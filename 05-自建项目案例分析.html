自建项目案例分析：Health Manager 智能用药提醒系统

本案例基于团队真实项目 HealthManager 智能用药提醒与健康管理系统（项目路径：/mnt/e/tools/visialstudio/project/project5）整理，面向软件工程课程考核/答辩，提供可复用的答题模板与延伸问答。

一、项目概述

1.1 项目背景与目标用户

随着我国老龄化程度加深，慢性病患者（如高血压、糖尿病、心血管疾病患者）的用药依从性成为影响疗效的关键因素。传统纸质药盒或单一闹钟提醒存在以下痛点：

  • 药名、剂量、服用时间容易记错；
  • 手动设置闹钟步骤繁琐，老年用户操作门槛高；
  • 普通闹钟在 Android 严格的后台省电策略下容易被延迟或漏报；
  • 家属无法远程掌握老人是否按时服药。

Health Manager 正是针对上述场景设计的 Android 原生应用，目标用户为：

  1. 中老年慢性病患者（核心用户）；
  2. 患者家属或照护者（协助管理用药计划）；
  3. 社区医生/药师（未来可扩展为远程随访场景）。

1.2 核心功能

功能模块 说明
拍照智能识别药品 用户对准药盒拍照，系统通过百度 OCR + 智谱 AI 自动提取药名、剂量、频次、配伍禁忌等信息。
用药提醒计划管理 支持 AI 自动生成、手动添加、编辑、删除用药闹钟，可设置每日/自定义周期。
高精度强提醒 基于 AlarmManager.setExactAndAllowWhileIdle + 前台服务 + TTS 语音播报，穿透 Doze 模式，锁屏也能全屏弹窗。
设备重启恢复 BootReceiver 监听开机广播，自动重新注册所有未到期闹钟，防止关机/清理后失效。
用药历史记录 打卡式历史足迹，支持按时间倒序查看，容量上限控制。

二、例题解析：软件系统建模

2.1 项目介绍与个人承担工作

在答辩或试卷中，常出现「请介绍你参与的项目，并说明你承担的工作」这类题目。下面给出 5 种角色的答题模板，可直接根据身份替换使用。

模板 A：组长 / 架构师（朱鹏远）

我担任 HealthManager 项目的组长和架构师。项目是为中老年慢性病患者打造的 Android 智能用药提醒 APP。我的核心工作是制定三层分层架构（表现层 / 业务逻辑层 / 数据持久层），设计模块间接口，封装 AlarmScheduler 统一调度系统闹钟，并实现 BootReceiver 完成设备重启后的闹钟自动恢复。在协作中，我采用「接口先行、实体驱动」的方式：先定义 MedicineAlarm 数据类和 scheduleAlarm(alarm) 方法签名，再让 UI 组、AI 组、持久化组并行开发，从而把 5 人代码像拼图一样拼装起来，降低了集成风险。

模板 B：视觉端 / AI 集成（吴轩赫）

我负责视觉输入和 AI 识别模块。用户点击「拍照识别」后，CameraCaptureActivity 调用 CameraX 获取药盒图像；我编写的 MedicineRecognitionEngine 在子线程中将图片压缩、Base64 编码，先请求百度 OCR 提取包装文字，再把文字连同用户已有药品列表一起投递给智谱 GLM-4-Flash，利用 Prompt 让大模型清洗出结构化的 {medicine, dosage, time} JSON。为避免网络阻塞主线程导致 ANR，整条链路放在子线程，并通过 ResultHandler 回调把结果安全传回主线程刷新 EditAlarmActivity 表单。

模板 C：后端进程保活（张恺祺）

我负责保证提醒在息屏、杀后台、甚至重启后仍能触发。技术上采用三层防护：调度层使用 AlarmManager.setExactAndAllowWhileIdle 穿透 Doze 休眠；执行层在 AlarmReceiver.onReceive() 中获取 WakeLock 并通过 startForegroundService() 拉起 AlarmService，挂上高优先级通知实现前台保活；灾备层通过 BootReceiver 监听 BOOT_COMPLETED，从本地存储读取闹钟并重新注册。服务返回 START_STICKY，异常终止后系统会尝试自动重启。

模板 D：数据持久化（周嘉晨）

我负责本地数据的存储与读取。核心实体是 MedicineAlarm，我用 AlarmStorageManager 单例将闹钟列表序列化为 JSON 数组，通过 SharedPreferences 以 MODE_PRIVATE 写入内部闪存；历史记录由 HistoryStorageManager 维护，采用 MAX_HISTORY_SIZE = 50 的 FIFO 溢出淘汰策略，防止数据无限膨胀。读取时加入 try-catch 容错：若 JSON 损坏则返回空列表并在下次保存时自动重建，避免应用闪退。

模板 E：UI 交互与测试（杨佳禾）

我负责全部前端界面和测试工作。表现层包括 MainActivity、CameraCaptureActivity、EditAlarmActivity、AlarmListActivity、HistoryActivity 及对应的 RecyclerView.Adapter。我采用 Material Design 组件和大字号、高对比度配色，降低老年用户操作门槛。同时我制定了《测试计划》和《测试分析报告》，组织 5 人交叉执行 14 项测试用例，覆盖功能、接口、弱网、边界、后台保活、重启恢复等场景，最终缺陷修复后再进行回归测试。

2.2 软件系统开发中常用的建模方法分类

软件系统建模的本质是用抽象化、可视化的手段描述系统「做什么」和「怎么做」。常见方法可分为四大类：

1. 结构化建模

以数据和过程为核心，适用于需求明确、数据关系清晰的系统。

方法 特点 适用范围
数据流图（DFD） 描述数据从输入到输出的变换过程，强调功能分解和数据流动。 需求分析、系统边界划分、模块接口设计。
数据字典 对 DFD 中所有数据流、数据存储、数据项进行精确定义。 与 DFD 配套使用，消除二义性。
E-R 图 描述实体、属性及实体间关系，是数据库设计的核心模型。 关系型数据库概念设计。

2. 面向对象建模

以对象和交互为核心，UML 是其标准表示法，适用于复杂、演化的系统。

图 特点 适用范围
用例图 从用户视角描述系统功能及 Actor 与 Use Case 的关系。 需求获取、功能范围确认。
类图 描述类的属性、方法及类之间的继承、关联、聚合、依赖关系。 静态结构设计。
顺序图 按时间轴展示对象间消息传递顺序。 动态行为设计、接口协议确认。
状态图 描述对象在生命周期内的状态转换及触发事件。 具有明显状态变化的实体，如闹钟、订单。
活动图 描述业务流程或算法的工作流，强调控制流和并行。 业务流程建模。

3. 形式化建模

使用严格的数学符号（如谓词逻辑、Z 语言、Petri 网、B 方法）描述系统行为和约束。

  • 特点：无二义性、可证明、高精度。
  • 适用范围：航空航天、金融交易、医疗设备等对安全性要求极高的关键系统。
  • 局限：学习成本高，与业务人员沟通困难，中小型应用较少使用。

例如，对闹钟触发的精确性可给出形式化规约： $$\forall a \in \text{Alarms}, \; |T_{actual}(a) - T_{scheduled}(a)| \le 100\,\text{ms}$$

4. 基于场景建模

通过具体使用场景描述系统行为，强调「谁在什么情况下想做什么」。

方法 特点 适用范围
用例（Use Case） 结构化描述用户与系统的交互步骤，含前置/后置条件、主/备选流。 需求文档、验收测试依据。
用户故事（User Story） 轻量级描述：「作为…，我希望…，以便…」。 敏捷开发、快速迭代。

2.3.2 数据流图：说明系统数据流转

说明：

  • 效果：DFD 让团队看清了数据在模块之间的传递路径，避免了 UI 层直接操作底层存储的越层调用，确保了三层架构的纪律性。

2.3.3 顺序图：拍照识别 → AI 解析 → 保存提醒

说明：

  1. 用户在 CameraCaptureActivity 拍照，生成 JPEG 图片路径；
  2. 调用 MedicineRecognitionEngine.recognize()，引擎在子线程中依次请求百度 OCR、智谱 AI；
  3. AI 返回 JSON 后，引擎通过 RecognitionCallback 通知 ResultHandler；
  4. ResultHandler 在主线程更新 EditAlarmActivity 表单；
  5. 用户确认后，数据写入 AlarmStorageManager 并调用 AlarmScheduler.scheduleAlarm() 注册系统闹钟。

效果：顺序图明确了跨线程回调机制和模块间消息顺序，为后续排查「网络请求白屏」「回调不到 UI」等问题提供了时序依据。

2.3.4 类图：说明三层架构的实体与关系

说明：

  • Entity 包：MedicineAlarm 核心数据类，携带 id、medicineName、dosage、time 等属性。
  • UI_View 包：所有 Activity 与 Adapter，负责渲染和事件响应。
  • Controller_Service 包：AI 引擎、调度器、广播接收器、前台服务，是业务逻辑中枢。
  • Data_Storage 包：两个单例 Manager 负责 JSON 持久化。
  • 关系：Manager 与实体为聚合；Activity 与引擎为依赖；Service/BroadcastReceiver 继承 Android 框架类。

效果：类图使 5 人团队对模块边界和依赖关系达成一致，避免了循环依赖和职责混乱。

2.3.5 状态图：闹钟从创建到触发到确认的状态转换

说明：

  • 闹钟实体经历「未创建 → 已保存（待触发）→ 已注册到系统 → 触发中（前台服务运行）→ 已确认服药 / 已忽略」等状态。
  • 触发事件包括：用户保存、系统时间到达、用户点击「我已服药」、用户点击忽略、设备重启后重新注册。
  • 效果：状态图帮助设计出完整的异常分支（如触发时 APP 被杀死怎么办？重启后如何恢复？），直接指导了 BootReceiver、START_STICKY、TTS 降级等代码实现。

3.2 软件设计类

Q1：采用了什么软件体系结构？

高分回答：

本项目采用三层分层架构（3-Tier Architecture）：

  1. 表现层（UI Layer）：Activity + Adapter，负责界面渲染和用户事件；
  2. 业务逻辑层（Logic Layer）：AI 引擎、调度器、广播接收器、前台服务，负责核心业务规则与系统协调；
  3. 数据持久层（Data Layer）：MedicineAlarm 实体 + AlarmStorageManager / HistoryStorageManager，负责数据存储。

上层依赖下层，下层不感知上层，通过 MedicineAlarm 实体和统一接口（如 scheduleAlarm()）交互，符合高内聚低耦合原则。

Q2：模块如何划分？高内聚低耦合体现在哪里？

高分回答：

模块划分对应概要设计中的 M-UI-01~M-UI-07、M-LGC-01~M-LGC-06、M-DAT-01~M-DAT-03。

高内聚：

  • AlarmScheduler 只做一件事：封装 AlarmManager 注册/取消/更新闹钟；
  • AlarmStorageManager 只负责闹钟 JSON 的读写；
  • MedicineRecognitionEngine 只负责 OCR + AI 的图像识别链路。

低耦合：

  • UI 层不直接调用 Android AlarmManager，而是通过 AlarmScheduler.scheduleAlarm(alarm) 黑盒调用；
  • AI 引擎通过 ResultHandler 回调与 UI 通信，引擎内部更换 OCR 厂商不影响 UI；
  • 数据实体 MedicineAlarm 作为通用介质，在 UI、逻辑、持久层之间传递，避免各层直接依赖彼此的具体实现。

Q3：为什么选择 SharedPreferences 而不是 SQLite/Room？

高分回答：

这是基于业务特性、数据规模和开发阶段的综合权衡：

  1. 数据量极小：闹钟通常 < 20 个，历史记录 ≤ 50 条，JSON 全量读写开销可忽略；
  2. 访问模式简单：基本是整存整取，没有复杂的多表关联、筛选、聚合查询；
  3. 开发效率高：SharedPreferences 无需定义 Schema、DAO、Migration，适合中期快速迭代；
  4. 读取速度快：50 条历史记录在本地闪存读取时间 < 1ms，满足「秒开」体验。

局限性：无并发锁保护、无云端备份、不支持复杂查询。后续若增加长期统计图表或多用户数据，可平滑迁移到 Room。

包图体现了纵向分层 + 横向职责的划分：ui 只依赖 engine/scheduler/data；data 不依赖 ui；service 通过 scheduler 和 data 完成恢复。

3.3 软件过程模型类

Q1：本项目适合用什么过程模型？

高分回答：

推荐使用迭代/增量模型 + 原型模型相结合：

  • 迭代/增量：把系统拆分为可运行的增量版本：
  • 第一轮：手动添加闹钟 + 基础提醒；
  • 第二轮：加入拍照识别 + AI 填表；
  • 第三轮：后台保活 + 重启恢复 + 历史记录。
  • 原型模型：在正式编码前快速做出可点击原型，邀请老年用户试用，验证交互逻辑和字体大小是否合适。

理由：

  1. 项目涉及外部 AI API，识别准确率和响应时间具有不确定性，需要快速试错；
  2. Android 后台保活策略在不同 ROM 上表现差异大，需要多次迭代适配；
  3. 5 人团队规模适中，短周期迭代便于及时调整分工。

Q2：为什么不完全用瀑布模型？

高分回答：

瀑布模型要求需求在前期完全确定，但本项目存在多处不确定性：

  • 百度 OCR + 智谱 AI 的实际识别效果在开发前难以精确评估；
  • 国产 ROM 对后台服务、精确闹钟的权限策略在不断变化；
  • 老年用户的真实操作习惯需要通过原型和测试逐步发现。

如果采用纯瀑布模型，很可能在后期集成阶段才发现 AI 解析不稳定或闹钟被系统拦截，导致大规模返工。迭代模型可以在每个增量结束后及时验证并修正方向。

Q3：敏捷方法在本项目中的体现

高分回答：

  • 小步快跑：每 1~2 周交付一个可演示增量；
  • 每日站会：5 人团队每天同步进度、阻塞点和接口依赖；
  • 角色自组织：根据模块接口划分任务，成员对自己的交付负责；
  • 持续集成：使用 Android Studio + Git 频繁合并，及时解决冲突；
  • 用户反馈驱动：通过原型和真机测试不断调整 UI 和提醒强度；
  • 迭代回顾：每轮测试后召开缺陷复盘会，形成 BUG 清单和改进建议。

Q2：单元测试、集成测试、确认测试分别测什么？

高分回答：

  • 单元测试：验证最小可测试单元的正确性。例如：
  • MedicineAlarm 的时间字符串解析是否正确；
  • AlarmStorageManager.serializeToJson() 和 getAlarms() 是否互逆；
  • AlarmScheduler 计算下一次触发时间是否考虑重复周期。
  • 集成测试：验证多个模块组合后的行为。例如：
  • 拍照后图片能否正确传入识别引擎；
  • AI 返回 JSON 后能否正确填充到 EditAlarmActivity；
  • AlarmReceiver 触发后能否正确启动 AlarmService。
  • 确认测试：验证软件是否满足用户需求。例如：
  • 用户是否能在 30 秒内完成拍照识别；
  • 闹钟在息屏、杀后台、重启后是否都能触发；
  • 老年用户是否能独立完成一次用药提醒设置。

Q4：性能/稳定性测试（闹钟触发精度、后台保活）

高分回答：

  • 闹钟触发精度：
  • 设置多个闹钟，用 Logcat 记录 AlarmReceiver.onReceive() 的实际触发时间；
  • 计算偏差 $\Delta t = t_{actual} - t_{scheduled}$，要求 $|\Delta t| \le 100\,\text{ms}$；
  • 分别在息屏、Doze、后台被杀场景下测试。
  • 后台保活：
  • 手动划掉应用进程，检查闹钟是否仍触发；
  • 开启「飞行模式+省电模式」测试服务行为；
  • 重启设备，验证 BootReceiver 是否恢复所有闹钟；
  • 使用 adb shell dumpsys alarm 查看 PendingIntent 注册状态。
  • 并发触发：
  • 同一时刻设置 5 个以上闹钟，验证通知不堆叠丢失、TTS 不冲突。

3.5 软件质量与维护类

Q1：如何保证提醒不漏报？

高分回答：

本项目采用多层冗余 + 系统级调度策略：

  1. 精确闹钟调度：使用 AlarmManager.setExactAndAllowWhileIdle()，可在 Doze 模式下强制唤醒 CPU；
  2. 前台服务执行：触发后通过 AlarmReceiver 立即拉起 AlarmService 并 startForeground()，挂上不可清除的高优先级通知，避免被系统回收；
  3. WakeLock 保护：在 AlarmReceiver 中获取 PARTIAL_WAKE_LOCK，确保关键逻辑执行期间 CPU 不睡眠；
  4. 服务自动重启：AlarmService 返回 START_STICKY，异常被杀后系统会尝试重建；
  5. 开机恢复：BootReceiver 监听 BOOT_COMPLETED，重启后重新注册所有闹钟；
  6. 用户引导：针对国产 ROM，引导开启「自启动」「电池优化白名单」「悬浮窗权限」等。

用可靠性语言可表示为：设各层失效概率为 $p_1, p_2, \dots, p_n$，则漏报概率为： $$P_{miss} = \prod_{i=1}^{n} p_i$$ 多层独立冗余使得 $P_{miss}$ 显著降低。

Q2：AI 识别错误的容错机制是什么？

高分回答：

医疗场景下，AI 错误可能导致严重后果，因此本项目坚持「AI 辅助、人工决策」原则：

  1. 人工二次确认：AI 返回的药名、剂量、时间只作为预填充，必须用户在 EditAlarmActivity 中确认后才能保存；
  2. 结果高亮提示：对 AI 填充字段使用醒目颜色，提醒用户重点核对；
  3. 手动输入降级：当网络不可用、API Key 失效或 OCR 失败时，自动跳转到手动输入界面，保证核心提醒功能不中断；
  4. JSON 格式校验：解析 AI 返回体时加入 try-catch，关键字段缺失时提示具体错误；
  5. Prompt 工程：在请求智谱 AI 时明确要求返回固定 JSON 格式，减少输出漂移。

Q3：如果后续要扩展功能（如家人远程提醒、医生端），如何演化？

高分回答：

基于现有三层架构，可按以下方式演化：

  1. 家人远程提醒：
    • 后端增加云端用户体系和推送服务（Firebase Cloud Messaging 或自建 MQTT）；
    • 患者端服药打卡后上传事件，家属端实时接收「已服药/未服药」通知；
    • 本地 HistoryStorageManager 可升级为 Room + 云端同步。
  2. 医生端：
    • 新增 Web 管理后台，医生可查看患者用药依从性报表；
    • 在 AI 分析阶段加入药品相互作用数据库，提升安全性评估准确性；
    • 通过 REST API 与 Android 端双向同步处方信息。
  3. 架构调整：
    • 数据层引入 Room 数据库处理多用户、长周期数据；
    • 业务层增加 Repository 模式，统一本地与远程数据源；
    • UI 层拆分为患者 APP、家属 APP、医生 Web 三端，共享核心业务逻辑模块。

3.6 软件工程实践类

Q1：5 人团队如何分工避免代码冲突？

高分回答：

  • 模块边界清晰：按三层架构将 5 人分配到 UI、AI、保活、持久化、测试五个模块，模块间通过接口和实体通信；

  • 每日集成：每天至少合并一次可编译代码，及时发现接口不匹配；
  • 代码评审：关键模块（如 AlarmScheduler、MedicineRecognitionEngine）由组长或相关成员 Review 后再合并；
  • 统一规范：命名规范、日志 TAG、错误码统一，便于后续测试和定位问题。

四、例题参考答案（对应 10 分综合分析题）

典型题目

请结合你参与的项目，说明： 1. 项目背景与主要功能； 2. 采用了哪些软件系统建模方法？请结合图示说明； 3. 项目采用了什么软件体系结构？如何体现高内聚低耦合？ 4. 在测试与质量保证方面做了哪些工作？

参考答案

1. 项目背景与主要功能

HealthManager 是为中老年慢性病患者设计的 Android 智能用药提醒应用。项目背景源于老年人漏服、错服药物的普遍问题，以及普通闹钟在 Android 后台限制下容易失效的痛点。主要功能包括：拍照识别药品（百度 OCR + 智谱 AI）、用药闹钟管理、高精度提醒（AlarmManager + 前台服务 + TTS）、设备重启恢复、用药历史记录。

2. 软件系统建模方法

本项目综合使用了面向对象建模和结构化建模：

  • 用例图：识别患者、Android 系统、OCR/AI 服务三个 Actor，明确拍照识别、提醒触发、历史查看等 Use Case；
  • 数据流图：描述从药品图像输入到提醒通知输出的完整数据变换过程；
  • 顺序图：刻画拍照 → OCR → AI → 回调 UI → 保存闹钟的时序关系，明确跨线程通信机制；
  • 类图：划分 Entity、UI_View、Controller_Service、Data_Storage 四个命名空间，展示三层架构的静态结构；
  • 状态图：描述闹钟从「待触发」到「触发中」再到「已确认」的状态转换，指导异常恢复设计。

3. 软件体系结构

采用三层分层架构：表现层（Activity/Adapter）、业务逻辑层（AI 引擎/调度器/服务/接收器）、数据持久层（Manager/Entity）。高内聚体现在每个模块职责单一，如 AlarmScheduler 只负责闹钟调度；低耦合体现在上层通过统一接口调用下层，如 UI 不直接操作 AlarmManager，而是调用 AlarmScheduler.scheduleAlarm(alarm)，AI 引擎通过回调与 UI 通信。

4. 测试与质量保证

按照《测试计划》执行了 14 项测试用例，覆盖功能测试、接口测试、集成测试、弱网容错、边界测试、后台保活、重启恢复等。关键质量指标包括：

  • 识别总耗时 $T_{total} = T_{compress} + T_{ocr} + T_{ai} + T_{parse} \le 35\,\text{s}$；
  • 闹钟触发偏差 $|\Delta t| \le 100\,\text{ms}$；
  • 历史记录容量 $M_{history} \le 50$；
  • 测试通过率 14/14，发现 8 项缺陷并给出改进建议。

通过以上工作，项目在中期阶段实现了核心功能的全链路打通，并具备了良好的可扩展性和可维护性。
