软件工程 — 全书重点速查

一、核心框架

软件工程的本质是“把软件视为产品、把软件开发视为工程”，在成本与进度约束下交付足够好的软件。全书围绕“需求 → 设计 → 实现 → 测试 → 维护”这一主线展开，核心手段是抽象、建模、模块化、重用、信息隐藏、关注点分离、双向追踪、工具辅助。

二、各章一句话重点

  1. 软件及其特点：软件 = 程序 + 数据 + 文档；是逻辑产品、设计开发而成、需求易变、系统复杂、缺陷隐蔽。
  2. 软件工程概述：诞生于 1968 年 NATO 会议，动机是解决软件危机；三要素为过程、方法学、工具；遵循抽象、建模、模块化、重用、信息隐藏等原则。
  3. 软件过程模型：瀑布、增量、迭代、原型、螺旋各有适用场景；选择模型需综合考虑项目特点、团队能力与模型优缺点。
  4. 软件需求基础：需求来自利益相关方，分为功能需求、质量需求、约束需求；需求工程是获取 → 分析 → 文档化 → 确认验证 → 管理的迭代过程。
  5. 获取软件需求：通过访谈、问卷、原型、头脑风暴、业务资料分析等方法导出需求；用例图是描述功能性需求的利器。
  6. 分析软件需求：从用例视角、行为视角、结构视角建模；常用 UML 图包括用例图、顺序图/通信图、类图、状态图。
  7. 软件设计基础：设计回答“如何做”；核心目标是满足需求并保证质量；关键原则是高内聚、低耦合、信息隐藏、关注点分离。
  8. 软件详细设计：在体系结构基础上精化用例设计、类设计、数据设计、子系统/构件设计；常用 UML 图包括活动图、状态图、顺序图、类图、构件图、包图。
  9. 编写代码：根据设计模型编码；缺陷 → 错误 → 失效；调试用于定位缺陷原因和位置。
  10. 软件测试：通过运行测试用例发现缺陷；测试层次为单元测试 → 集成测试 → 确认测试 → 系统/非功能测试；测试方法分为白盒与黑盒。
  11. 软件维护与演化：维护分为纠正性、适应性、完善性、预防性；软件会逻辑老化，重构是恢复可维护性的关键手段。

三、重点细讲

3.1 五种软件过程模型对比

模型 指导思想 关注点 适合软件 管理难度
瀑布模型 按阶段顺序推进 与软件生命周期一致，每阶段需评审 需求明确、变动小的项目 低
增量模型 分增量渐进交付 快速交付部分功能，可并行开发多个增量 需求较明确、可模块化的项目 中等
迭代模型 多次迭代完善 每次迭代是完整小周期，体现“小步快跑” 需求不确定、持续变化的项目 中等
原型模型 以原型导出需求 通过原型与用户交流、持续确认需求 需求难以表述、获取困难的项目 较低
螺旋模型 风险驱动的迭代 集成迭代与原型，引入风险分析 风险高、需求变化大的大型项目 高

选择建议：需求稳定选瀑布/增量；需求模糊易变选迭代/原型；风险高选螺旋。

3.2 黑盒测试与白盒测试对比

对比维度 黑盒测试 白盒测试
依据 软件外部功能与性能描述 程序内部结构与执行流程
是否需了解代码 不需要 需要
测试对象 功能、接口、输入输出 语句、分支、路径、条件
典型方法 等价类划分、边界值分析、决策表、因果图 语句覆盖、分支覆盖、路径覆盖、基本路径测试
优点 与实现无关，可提前设计；适合验证需求 能深入发现内部逻辑错误
典型应用阶段 集成测试、确认测试、系统测试 单元测试
关系 关注“做什么” 关注“怎么做”

3.3 软件测试层次对比

测试阶段 测试对象 测试方法 主要依据 实施时机
单元测试 函数、过程、方法、类 白盒测试为主 详细设计文档 详细设计阶段制定计划，编码后执行
集成测试 模块之间的接口与交互 黑盒测试为主 概要设计文档 概要设计阶段制定计划
确认测试 软件功能与性能是否满足需求 黑盒测试 软件需求规格说明书 需求分析阶段制定计划
系统测试/非功能测试 整个系统及其运行环境 黑盒/性能/安全/强度测试等 需求规格说明与运行环境要求 集成完成后
回归测试 修改缺陷后的新版本 重运行已有用例 历史测试用例 缺陷修复后

补充： - 单元测试需要驱动程序和桩模块。 - 集成策略包括自顶向下和自底向上。 - 确认测试包括 α 测试（开发方内部模拟用户）和 β 测试（典型用户在实际环境使用）。

3.4 等价类划分 vs 边界值分析

方法 核心思想 适用条件 用例设计原则
等价类划分 将输入集合划分为若干等价类，每类取一个代表值测试 输入域可分类 范围：有效类、大于最大值、小于最小值；值：有效、大于、小于；集合：在集合内/外；布尔量：真/假
边界值分析 软件易在边界处出错，重点选取边界及其邻近值 输入存在范围或边界 范围 $(a,b)$：取 $a,b$ 及紧邻左右的值；一组数：取最大、最小、次大、次小；数据结构有界时检查其边界

关系：边界值分析通常作为等价类划分的补充，二者常结合使用。

3.5 需求工程活动

需求工程的一般性过程：

$$ \text{获取} \rightarrow \text{分析} \rightarrow \text{文档化} \rightarrow \text{确认和验证} $$

全程由软件需求管理（变更管理、追溯管理、配置管理）支撑。

需求获取方法：访谈、会议、问卷、现场观摩、业务资料分析、软件原型、头脑风暴、群体化方法、大模型生成等。

需求分析三视角：

视角 关注的问题 UML 图
用例视角 具有哪些功能、功能间关系、与利益相关方关系 用例图
行为视角 用例如何通过对象交互来达成 顺序图、通信图、状态图
结构视角 业务领域概念及相互关系 类图

3.6 软件生命周期

软件生命周期是指软件从提出开发、开发实现、运行维护到最终退役的全过程。以瀑布模型为例，典型阶段如下：

阶段 核心任务 主要产出
需求分析 定义功能需求与非功能需求 软件需求模型、需求规格说明书、确认测试计划
概要设计 建立软件总体架构 概要设计模型、概要设计文档、集成测试计划
详细设计 设计模块内部细节（算法、数据结构） 详细设计模型、详细设计文档、单元测试计划
编码实现 编写程序代码并进行单元测试 经过单元测试的源程序代码
集成测试 组装模块，发现接口缺陷 集成测试报告
确认测试 从用户角度验证是否满足需求 确认测试报告

四、必背公式与定义

  • 软件 = 程序 + 数据 + 文档

$$ \text{软件} = \text{程序} + \text{数据} + \text{文档} $$

  • 软件开发 = 软件创作 + 软件生产

$$ \text{软件开发} = \text{软件创作} + \text{软件生产} $$

  • 测试用例四元组：

$$ (\text{输入数据},\ \text{前置条件},\ \text{测试步骤},\ \text{预期输出}) $$

  • Cyclomatic 复杂度（环形复杂度）：

$$ V(G) = E - N + 2 $$

其中 $E$ 为流图边数，$N$ 为流图节点数；或简化为 $V(G) = P + 1$，$P$ 为判定节点数。

  • 维护工作量模型：

$$ M = P + K \cdot e^{(c-d)} $$

其中 $P$ 为生产性工作量，$K$ 为经验常数，$c$ 为复杂度，$d$ 为熟悉程度。

  • 缺陷链：缺陷 → 错误 → 失效
  • 模块设计原则：高内聚、低耦合
  • 面向对象设计 6 大原则：SOLID + 迪米特法则
  • S：单一职责原则（Single Responsibility Principle）
  • O：开闭原则（Open/Closed Principle）
  • L：里氏替换原则（Liskov Substitution Principle）
  • I：接口隔离原则（Interface Segregation Principle）
  • D：依赖倒置原则（Dependency Inversion Principle）
  • LoD：迪米特法则 / 最少知识原则（Law of Demeter）

五、高频考点与应试提示

5.1 高频简答题

  1. 软件危机：在计算机软件开发和维护过程中遇到的一系列严重问题；表现为开发成本高、进度难以控制、质量难以保证、维护困难、失败风险大；原因是对软件复杂性和特殊性认识不清、缺乏有效开发方法、缺乏成功经验。
  2. 软件生命周期各阶段任务：需求分析、概要设计、详细设计、编码实现、集成测试、确认测试（见 3.6）。
  3. 顺序图与协作图/通信图区别：二者同属 UML 交互图、语义等价、可相互转换；顺序图强调消息传递的时间顺序，协作图强调对象间的合作关系与结构。
  4. 模块独立性度量：内聚度（模块内部结合紧密程度，越高越好）和耦合度（模块之间相关程度，越低越好）；设计原则为高内聚、低耦合。
  5. 测试步骤及对象：单元测试（模块内部）、集成测试（模块接口）、确认测试（功能与性能）、系统测试（整体系统）、回归测试（修改后的版本）。
  6. 建模方法分类：结构化建模（DFD、数据字典）、面向对象建模（UML）、基于构件建模、形式化建模、原型建模、基于场景建模；需能简述特点与适用范围。

5.2 高频设计/分析题

  1. 画数据流图并转化为软件结构图

    • 先画 0 层（上下文图）和 1 层 DFD，注意外部实体、转换、数据流、数据存储。
    • 区分变换型与事务型数据流图：变换型有输入流-变换中心-输出流；事务型有事务中心+多条动作路径。
    • 事务型映射为“接收路径控制 + 事务分析/散转 + 各动作路径模块”的层次结构。

  2. 画用例图与顺序图

    • 用例图：明确 Actor、Use Case、系统边界、关系（包含 include、扩展 extend、继承 inherit）。
    • 顺序图：从左到右依次为 Actor → 边界类 → 控制类 → 实体类 → 外部接口边界类；消息按时间自上而下传递，注意生命线、活跃期、同步/异步/返回消息图符。

  3. 程序流程图 + 环形复杂度 + 路径覆盖

    • 正确绘制程序流程图。
    • 转换为控制流图，计算 $V(G) = E - N + 2$ 或 $V(G) = P + 1$。
    • 根据基本路径设计测试用例，覆盖所有独立路径。

  4. 等价类划分 + 边界值分析

    • 按范围、特定值、集合、布尔量划分有效/无效等价类。
    • 边界值重点选取边界点及其左右邻近值，结合非法格式用例。

5.3 应试提示

  • 先审题再画图：看清题目要求结构化方法还是面向对象方法，画几层图，是否要求转化为结构图。
  • 图符规范：UML 图符（Actor 火柴人、用例椭圆、类三层矩形、菱形聚合/组合、空心三角继承等）要画对。
  • 路径覆盖用例：每个用例需给出输入、执行路径、预期输出，避免只写输入。
  • 对比题：从“定义、关注点、依据、方法、阶段、优缺点”等维度分条作答。
  • 不要混淆概念：
  • 增量模型是“一块一块地做”，迭代模型是“一遍一遍地改”。
  • 测试的目的是发现缺陷，不是证明无缺陷。
  • 软件不会物理磨损，但会逻辑老化。

提示：本文档为考前速查，建议对照教材课件中的各类示意图进行手绘练习，确保能在考场上独立完成数据流图、结构图、用例图、顺序图和程序流程图的绘制。
