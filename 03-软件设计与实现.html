第7-9讲：软件设计与实现

对应课件：3.1 软件设计基础、4.1 软件详细设计、5.1 编写代码
复习目标：掌握“软件设计”在需求分析与编码实现之间的承上启下作用，熟悉结构化与面向对象两类核心设计方法，能够绘制/阅读 DFD、结构图、UML 详细设计图，并理解编码、缺陷、调试的基本概念与实践。

一、知识点讲解

3.1 软件设计基础

1. 何为软件设计

软件设计是在软件需求的基础上，综合考虑各种制约因素，探究并给出软件实现解决方案的过程。它回答的是“如何做（How）”的问题，而需求分析回答的是“做什么（What）”。

  • 设计前提：软件需求。需求定义了要做什么样的软件。
  • 设计考虑：制约因素，包括
  • 资源：时间、人力、财力、开发辅助工具；
  • 技术：技术平台选择，例如采用 DBMS 还是文件系统等。

软件系统的解决方案类似“实现图纸”，通常要描述：

  • 模块的组织方式；
  • 每个模块的功能与接口；
  • 模块之间的交互；
  • 模块内部的算法；
  • 人机交互的界面和方式；
  • 数据结构、数据库设计与组织等。


软件设计处于需求分析与编码实现之间，是二者之间的“桥梁”。

2. 需求分析、软件设计、软件实现之间的关系

阶段 关注点 主要产出
需求分析 要做什么，明确问题和目标 软件需求模型和文档
软件设计 如何做，绘制实现图纸 软件设计模型和文档
编码实现 做出来，开展施工 源程序代码

设计的好坏直接决定最终产品的质量。设计阶段必须确保：

  • 对需求的正确理解；
  • 在技术平台和资源约束下的可行性；
  • 设计元素之间的一致性与无冲突。

3. 设计的多样性及其质量要求

软件设计是一个创造性过程。同一个需求往往可以产生多种设计方案，因此需要有一套质量标准来区分优劣。

软件设计的质量要求：

  1. 正确性：正确实现所有软件需求项；设计元素之间无逻辑冲突；在可用资源和技术平台约束下可完整实现。
  2. 充分性：所有设计元素已充分细化，模型易于理解，编程人员无需再面对影响功能和质量的技术抉择。
  3. 优化性：以合理、充分优化的方式实现需求，表现出良好的质量属性，尤其是正确性、有效性、可靠性和可修改性。
  4. 简单性：模块职责简明易懂，模块间关系简单直观，结构尽可能自然反映问题结构。

高质量软件设计还强调：可靠性、可维护性、可重用性、可追踪性、可移植性、可互操作性、有效性、安全性等。评价质量时应从用户、开发者、维护人员等多个利益相关者角度出发。

4. 软件设计过程

典型的软件设计过程包含以下主要活动（可迭代）：

软件需求
   │
   ▼
体系结构设计 ──► 人机交互设计 ──► 软件详细设计 ──► 文档化 ──► 软件设计评审
   │                  │                │
   └──────────────────┴────────────────┘
              软件设计管理（变更、追溯、基线）

  1. 软件体系结构设计

    • 从全局、宏观视角设计系统；
    • 确定构成要素（函数、方法、类、程序包）及其关系（依赖、交互）；
    • 职责分派、接口定义、交互协作行为；
    • 每个模块被视为“黑盒子”，关注可扩展、可维护、可重用、可移植、可互操作等。

  2. 用户界面设计

    • 设计系统对外展示以及与用户交互的界面；
    • 明确输出（系统告知用户的信息）与输入（需要用户提供的信息）；
    • 关注直观、友好、易于操作和理解。

  3. 软件详细设计

    • 对体系结构设计和用户界面设计进行细化、精化；
    • 获得面向实现的高质量设计模型；
    • 包括构件/类设计、接口设计、算法设计、数据设计等。

  4. 撰写设计文档

    • 按照软件设计规格说明书的规范记录设计信息。

  5. 软件设计评审

    • 验证设计是否实现需求、分析设计质量、发现缺陷并协商解决。

  6. 软件设计管理

    • 变更管理、追溯管理、基线管理。

5. 三类软件设计元素

元素 说明 示例
过程、函数、设计类 最基本的设计单元和模块单元 类中的方法、全局函数
软构件 可分离、可独立部署和执行、可单独重用 .DLL、Java JAR 包、微服务镜像
子系统 完成特定功能、逻辑相关的一组模块集合 软件包、某个业务子系统

元素之间的关系：一个软件系统由若干子系统组成；子系统可包含软构件、设计类/函数；软构件也可由更小的软构件、类和函数组成；所有设计元素在编码时都有对应物。

6. 软件设计要考虑的因素与设计原则

设计时需要考虑：

  • 满足需求：正确、一致、可行、完整、无冗余；
  • 权衡抉择：比较多种设计方案的优缺点，综合考虑成本、进度、质量等因素；
  • 应对变化：易于理解、扩展和高效处理变更。

软件设计的基本原则：

  1. 抽象与逐步求精

    • 抽象是忽略与当前目标无关部分、聚焦相关方面的策略；
    • 软件设计是从高层抽象（体系结构）到低层抽象（详细设计）逐步过渡的“逐步求精”过程。

  2. 模块化、高内聚度、低耦合度

    • 将系统分解为一组相对独立的模块，每个模块实现单一功能；
    • 内聚度（Cohesion）：模块内部各成分之间彼此结合的紧密程度，越高越好；
    • 偶然性内聚 → 逻辑性内聚 → 时间性内聚 → 过程性内聚 → 通讯性内聚 → 顺序性内聚 → 功能性内聚（最高）。
    • 耦合度（Coupling）：模块之间的相关程度，越低越好；
    • 非直接耦合 → 数据耦合 → 控制耦合 → 特征耦合 → 外部耦合 → 公共耦合 → 内容耦合（最强、应避免）。
    • 好的设计追求“模块内部强内聚、模块之间低耦合”。

  3. 信息隐藏

    • 模块应把内部信息对不需要知道的模块隐藏，只交换必要信息；
    • 优点：增强模块独立性、支持并行开发、减少错误传播、便于测试维护、便于增加新功能；
    • 面向对象中通过 public/private/protected 实现。

  4. 关注点分离

    • 关注点是对概念、任务和目标某个侧面的聚焦（如结构、行为）；
    • 将不同性质的关注点分离开，在适当时间分别处理，再整合；
    • 避免“胡子眉毛一把抓”。

  5. 软件重用

    • 尽量重用已有软件资产，并使自己开发的系统易于重用；
    • 可重用资产：代码片段、函数、类、构件、开源软件、设计模式、开发知识等；
    • 技术手段：封装、接口、继承、多态等。

  6. 迭代设计

    • 设计不是一次完成，需要反复精化、持续优化。

  7. 可追踪性

    • 设计应能追溯到分析模型，需求能在设计中找到对应实现。

其他常用原则还包括：经常关注架构、数据设计与功能设计并重、必须设计接口、用户界面符合最终用户要求、设计表述易于理解等。

7. 软件设计方法

软件设计方法明确了设计所依据的思想、理念、技术、过程和描述语言。

7.1 结构化软件设计

产生于 20 世纪 70 年代，代表性成果是面向数据流的软件设计方法学。基本思想：

  • 基于结构化需求分析结果（数据流图 DFD）；
  • 设计为以功能模块为核心的层次化软件体系结构；
  • 最终由结构化程序设计语言（如 C、Fortran）实现。

输入：需求分析阶段得到的数据流图。
输出：层次化的软件体系结构图。
核心方法：将数据流图映射为软件体系结构图。

数据流图有两种典型形式：

  • 变换型数据流图：由输入流、变换流（事务处理中心）、输出流组成；
  • 事务型数据流图：由事务中心、接收路径和若干条动作路径组成。

变换型映射方法：

  1. 确定输入流、输出流和变换流：划定输入流界和输出流界；
  2. 一级分解：导出三层结构——顶层主控模块、中间层三个控制模块（输入流控制、变换流控制、输出流控制）、底层对应各转换映射来的模块；
  3. 二级分解：把输入流、变换流、输出流中的每个转换映射为底层模块，并置于相应控制模块下；
  4. 优化：删除不必要模块、合并高耦合模块、拆分低内聚模块。

事务型映射方法：

  1. 确定事务中心和动作路径：将事务型 DFD 分为事务中心、接收路径、动作路径集合；
  2. 一级分解：映射为“接收路径控制 + 散转”结构，顶层为事务控制；
  3. 处理事务中心和每条路径：把事务中心及每条动作路径的数据流图映射为相应模块；
  4. 优化：提高模块化程度和质量。

结构化设计方法的特点：以数据流为输入、层次化体系结构为输出，采用转换/映射方法，属于体系结构设计工作，可在此基础上继续详细设计。

7.2 面向对象软件设计方法学

针对面向对象需求分析所得到的模型（用例图、交互图、分析类图）进行不断精化（而非简单转换），获得子系统、构件、设计类等设计元素，形成不同视角、不同抽象层次的设计模型。

典型过程：软件体系结构设计 → 用户界面设计 → 用例设计 → 构件/子系统设计 → 数据设计 → 类设计 → 详细设计。

常用 UML 图：

  • 包图：表示软件体系结构；
  • 顺序图：表示对象之间的交互时序及内容。

优势：

  • 高层抽象和自然过渡，贴近现实世界；
  • 多种形式和粗粒度的软件重用；
  • 系统化支持体系结构、界面、数据、构件、子系统、用例、类设计等；
  • 通过接口、抽象类、继承、实现等机制支持扩展和变更。
7.3 软件设计的 CASE 工具
  • 文档撰写工具：Microsoft Office、WPS 等；
  • 模型绘制工具：Microsoft Visio、StarUML、ArgoUML 等；
  • 分析与转换工具：IBM Rational Rose 等；
  • 配置管理工具：Git、GitHub、GitLab、PVCS、SourceSafe 等。
7.4 软件设计工程师

负责体系结构设计、数据设计、用户界面设计、详细设计等，角色包括架构师、数据设计工程师、UI 设计工程师等。需要具备创新能力、抽象与建模能力、质量保证能力、组织协调沟通能力、权衡抉择能力。

8. 软件设计输出及评审

软件设计的输出：

  • 软件设计模型：从多个视角、不同抽象层次描述设计信息，常用 UML、模块图、层次图等；
  • 软件设计文档：采用自然语言结合设计模型，详细记录体系结构、子系统/构件、界面、用例、数据等设计。

软件设计文档常见内容：

  1. 引言（编写目的、读者对象、系统概述、文档概述、定义、参考资料）；
  2. 软件设计约束和原则；
  3. 软件设计方案（体系结构、用户界面、用例、子系统/构件、类、数据、部署等）；
  4. 实施指南。

软件设计中的常见缺陷：

  • 设计不满足需求；
  • 设计质量低下；
  • 设计存在不一致；
  • 设计不够详尽。

软件设计评审：

  • 目的：发现设计模型和文档中的缺陷；
  • 参与者：设计工程师、程序员、测试工程师、用户、质量保证人员等；
  • 评审内容：文档规范性、可理解性、合法性、质量水平、需求满足度、优化空间等。

软件设计管理：

  • 变更管理：明确变更范围、影响、版本变化；
  • 追溯管理：搞清楚变更原因，评估影响域；
  • 基线管理：通过评审后纳入基线库。

4.1 软件详细设计

1. 详细设计概述

详细设计是体系结构设计与编码实现之间的“桥梁”。它在体系结构设计和用户界面设计的基础上，进一步细化、精化，获得面向实现、可直接支持编码的高质量设计模型。

输入：

  • 软件需求；
  • 体系结构设计模型；
  • 用户界面设计模型；
  • 实现约束和限制。

输出：软件详细设计模型 + 软件详细设计规格说明书。

设计关注点的变化：

体系结构设计 详细设计
结构性 过程性
全局性 局部性
关键性 细节性
粗粒度 细粒度

详细设计过程（四个主要活动）：

用例设计 ──► 类设计 ──► 数据设计 ──► 子系统和构件设计
                │
                ▼
          文档化与评审详细设计

详细设计的要求和原则：

  • 针对软件需求：每项需求都有相应设计元素实现；
  • 深入优化设计：提升性能、可靠性、可重用性、可维护性；
  • 设计足够详细：足以支持程序员编码；
  • 充分软件重用：从多个维度和层次重用已有资产，降低成本。

2. 描述详细设计的 UML 图

视点 图 说明
结构 包图 从包层面描述系统静态结构
结构 类图 从类层面描述系统静态结构
结构 对象图 从对象层面描述系统静态结构
结构 构件图 描述系统中构件及其依赖关系
行为 状态图 描述状态的变迁
行为 活动图 描述系统活动的实施，可并发
行为 通信图 描述对象间的消息传递与协作
行为 顺序图 描述对象间的消息传递时序
部署 部署图 描述工件在物理运行环境中的部署
用例 用例图 从外部用户角度描述系统功能
2.1 活动图

活动图描述实体为完成某项功能而执行的操作序列，其中某些操作或其子序列可能存在并发和同步。

主要构成：

  • 活动节点：表示计算过程；
  • 决策节点：根据条件进行活动决策；
  • 边：控制流（触发后续操作）或信息流（操作间信息交换）；
  • 并发控制：
  • 分叉（Fork）：一条控制流分成多条可并行执行的控制流；
  • 汇合（Join）：多条并行控制流同步合并为单条控制流；
  • 泳道（Swimlane）：将活动图按负责对象或线程划分为若干区域，每个活动节点位于负责执行它的对象/线程所在区域，可清晰表达职责、分工、协同与同步。

绘制原则：

  • 从决策点出发的每条边均应标注条件，且条件覆盖完整、互不重叠；
  • 分叉与汇合节点必须匹配；
  • 一张活动图可浓缩为另一活动图中的单个活动节点，前者称为子活动图，后者称为父活动图。

3. 用例设计

需求分析阶段的用例描述功能；详细设计阶段要明确这些功能是如何通过一组设计元素（子系统、构件、设计类、界面类）协同实现的。

任务：

  • 针对每个用例，基于体系结构和用户界面设计元素，设计软件实现方案；
  • 描述各设计元素如何通过协作完成用例；
  • 产生详细设计信息，如新的设计元素、交互、细节、关联；
  • 输出用例实现的顺序图、设计类图等。

原则：

  • 以软件需求为基础；
  • 通过整合前期设计元素来实现用例；
  • 在用例实现过程中进一步精化设计。

过程：

  1. 设计用例实现方案

    • 基础：分析阶段用例交互图、体系结构/界面设计元素、各设计元素职责；
    • 方法：将分析类转化为设计类，引入体系结构/界面设计元素，形成用例实现的交互模型；
    • 结果：用例实现的顺序图，引入构件、子系统、类等，精化接口、方法、交互。

  2. 构造设计类图

    • 从分析类图、体系结构图、用例实现顺序图中寻找初始设计类；
    • 确定设计类的职责、类间关系、属性；
    • 遵循模块化、职责单一原则调整和优化。

    分析类到设计类的精化常见情形：

    • 一个分析类职责由某个设计类单项操作完整实现；
    • 一个分析类职责由某个设计类的多项操作实现；
    • 一个分析类职责由多个设计元素协同完成。

  3. 优化和评审用例设计方案

    • 尽可能重用已有资产；
    • 合并职责相同或相似的设计元素；
    • 借助继承或代理机制重组设计元素；
    • 确保各用例实现方案和谐共存、无逻辑冲突。

输出：用例设计的顺序图、用例设计类图。

4. 类设计

类设计对界面类、关键设计类、设计类进行优化和精化，明确内部实现细节，使其可以直接提交软件实现。

输入：体系结构设计模型、用户界面模型、用例模型、用例交互模型、分析类模型、用例设计模型。

输出：类设计模型（类图、状态图、活动图等）。

精化到什么程度：

  • 类的可见范围；
  • 属性的数据类型；
  • 方法的实现算法；
  • 类间关系；
  • 状态变化或活动情况。

类设计原则：准确化、细节化、一致性，并遵循模块化、高内聚、低耦合、信息隐藏等设计基本原则。

类设计过程：

  1. 确定类的可见范围

    • public：系统中所有包可见；
    • protected：所在包及子类可见；
    • private：仅所在包可见。
    • 原则：尽量缩小可见范围，将类“隐藏”于包内。

  2. 精化类间关系

    • 语义强度从高到低：继承 > 组合 > 聚合 > 普通关联 > 依赖；
    • 遵循“自然抽象”原则和“强内聚、松耦合”原则，尽量采用语义连接强度较小的关系；
    • 明确多重性：1:1、1:n、0:n、n:m 等。

  3. 精化类属性和方法

    属性设计： - 用业务领域名词命名； - 可见范围尽量小，遵循信息隐藏； - 结合类关系选择引用/集合类型： - 1:1 关联/聚合（非组合）→ 指针或引用； - 1:n 关联/聚合（非组合）→ 元素为指针/引用的集合； - 1:1 组合 → 类型为 B 的属性； - 1:n 组合 → 元素类型为 B 的集合。

    方法设计：细化方法名称、参数表、返回类型、作用范围、功能描述、实现算法、前置条件（pre-condition）、后置断言（post-condition）等。

    特殊方法：对象创建（初始化）、对象删除（释放资源）、对象比较、对象复制。

  4. 构造状态图和活动图

    • 状态图：适用于对象生命周期中有复杂状态变迁；
    • 活动图：适用于某职责需要执行一系列方法、与多个对象交互。

  5. 评审和优化类设计

    • 按强内聚、松耦合拆分/合并类和方法；
    • 判断是否足以支持编码；
    • 判断类间关系是否自然；
    • 判断可见范围、属性/方法作用范围是否合适。

类设计输出：

  • 详细的类属性、方法、类间关系类图；
  • 描述方法实现算法的活动图；
  • 必要时的状态图。

5. 数据设计

数据设计负责将软件涉及的信息抽象为计算机可理解和持久保存的数据，并设计其存储、组织和访问方式。

任务：

  • 设计需要持久保存的数据及其关系；
  • 设计数据组织方式（如关系数据库中的表、关键字、外键）；
  • 为提高性能而设计持久存储机制优化设施。

数据设计过程：

  1. 确定持久数据；
  2. 确定数据存储和组织方式；
  3. 设计数据访问操作；
  4. 评审和优化数据设计。

持久化映射（面向对象模型 → 关系数据库）：

OO 概念 关系数据库概念
类 表（table）
对象 记录（record）
属性 字段（field）

关联关系映射：

  • 1:1 / 1:n 关联：将一端表的关键字段纳入另一端表作为外键即可表示；
  • n:m 关联：引入交叉表，将两端表的关键字段作为外键存入交叉表。

继承关系映射：

  • 方案一：将父类所有字段复制到子类表。优点查询快，缺点浪费空间、易产生数据不一致；
  • 方案二：仅将父类关键字段作为外键纳入子类表。优点避免冗余，缺点读取时需联合查询。

数据操作：写入、查询、更新、删除四类基本操作，以及数据验证操作。

评审：正确性、一致性、时空效率、可扩展性。

输出：描述数据设计的类图、描述数据操作的活动图。

6. 子系统和构件设计

体系结构设计和用例设计引入了子系统或构件，需要从封装和重用角度对其进行深入设计。

子系统设计：

  • 确定子系统内部结构；
  • 设置更小粒度的子系统、构件、设计类；
  • 明确它们之间的协作关系；
  • 确保协同实现子系统接口规定的功能和行为。
  • 输出：包图、构件图、顺序图、活动图、类图。

构件设计：

  • 定义构件内部的设计元素及其协作方法；
  • 内部元素可以是子构件或粒度更细的类；
  • 构件是可独立部署和运行的设计元素。

子系统设计过程：

  1. 精化子系统内部设计元素：明确构成元素、职责、协作方式；
  2. 构造子系统的设计类图：从顺序图中抽象出类、方法、关系；
  3. 构造子系统的状态图和活动图（必要时）；
  4. 评审子系统设计。

设计类图推导方法：

  • 顺序图中对象对应的类抽象为设计类；
  • 对象 a 向 b 发消息 m(p) → 类 B 有处理 m 的方法和存储 p 的属性；
  • 根据消息确定类间关系：消息传递对应关联或依赖；通过接口交互对应依赖；一般/特殊关系对应继承。

子系统设计评审：完整性、设计质量、可满足性、正确性、一致性。

7. 详细设计文档化与评审

输出：

  • 用 UML 类图、构件图、包图、状态图、顺序图等描述的详细设计模型；
  • 软件详细设计规格说明书。

设计整合：汇总体系结构、界面、用例、子系统/构件/类、数据等模型，形成完整方案。

设计验证：验证正确性、优化性、充分性、一致性、冗余性，发现并整改问题。

设计文档评审人员：用户、设计人员、程序员、需求分析人员、质量保证人员、测试工程师、配置管理工程师。

评审设计文档的准则：规范性、简练性、正确性、可实施性、可追踪性、一致性、高质量。

评审步骤：阅读汇报 → 发现收集问题 → 讨论达成一致并修改 → 纳入配置。

5.1 编写代码

1. 编写代码的任务

编写代码是根据软件设计信息，借助程序设计语言，编写出目标软件系统源程序代码，并开展程序单元测试、代码审查、程序调试等质量保证工作的过程。

  • 既是创作过程，也是生产过程；
  • 输入：软件体系结构设计模型、用户界面设计模型、软件详细设计模型、编程语言与支撑工具；
  • 输出：高质量的程序代码。

2. 编写类代码

根据设计类图，逐项落实以下关系与机制：

  1. 编写实现类的代码：将可见性、类名、属性、方法转换为语言结构；
  2. 编写实现类方法的代码：基于活动图描述的方法算法编写方法体；
  3. 编写实现类间关联的代码：根据方向性、多重性、角色名、约束等实现；
  4. 编写实现聚合和组合关系的代码：类似关联，根据多重性设计属性数据结构；
  5. 编写实现接口关系的代码：使用语言提供的 implements 等机制；
  6. 编写实现继承关系的代码：使用 extends 等机制；注意 Java 单继承、C++ 多继承；
  7. 编写实现包的代码：用 package 组织类，对应代码目录结构。

3. 编写用户界面代码

  • 根据界面设计模型定义界面类属性（静态元素、输入元素、命令元素等）；
  • 编写界面类方法，处理界面操作和事件响应；
  • 实现界面之间的跳转关系。

4. 编写数据设计代码

  • 创建数据库关系表及字段，满足设计约束；
  • 编写增删改查等数据库操作代码；
  • 实现连接/关闭数据库、判断数据合法性等辅助代码。

5. 代码片段的重用

代码片段对应类代码中实现某一细粒度功能的一组语句序列。程序员可以从 Stack Overflow、CSDN 等开源技术问答社区中寻找经过实践检验的代码片段，经过理解、选定和适配后重用于自己的程序。

6. 基于大模型的代码推荐和生成

  • 与大模型工具（如 ChatGPT）或 CASE 工具（如 Copilot）交互生成代码；
  • 采用结对编程形式，分解任务、提供清晰提示；
  • 必须对生成代码进行鉴别、分析和测试，不可盲目使用。

7. 软件缺陷、错误与失效

软件缺陷：软件制品中存在不正确的描述和实现，可存在于需求、设计、代码中，产生于软件开发全过程。

  • 错误（Error）：存在缺陷的程序代码在运行时产生不正确或不期望的内部状态；
  • 失效（Failure）：运行错误的程序无法为用户提供所需功能和行为。

关系链：缺陷 → 错误 → 失效。缺陷是根源，错误是内部表现，失效是外部表现。

8. 软件缺陷的描述

一个缺陷报告通常包含：

  • 唯一标识符；
  • 类型（需求/设计/代码缺陷，代码缺陷又可分为逻辑、计算、判断等）；
  • 严重程度（危急、严重、一般、轻微）；
  • 症状与运行表现；
  • 修复优先级；
  • 处理状态；
  • 发现者、发现时机、源头、原因等。

9. 软件缺陷的应对方法

  • 预防缺陷：运用软件工程技术、管理手段、结对编程、培训、CASE 工具等减少缺陷；
  • 容忍缺陷：通过容错机制允许软件出错仍能正常运行，常用于高可靠系统；
  • 发现缺陷：评审、走查、测试等手段发现潜藏缺陷；
  • 修复缺陷：通过调试定位缺陷原因和位置，修改代码并回归测试。

缺陷状态：尚未确认、有效、无效、重复、已分配、已修复、信息不完整、已解决、已关闭。

10. 程序调试

调试是基于程序代码，确定软件缺陷原因、定位缺陷位置，从而知道“哪里错了、如何修复”的过程。

调试步骤：

理解缺陷及其症状
      │
      ▼
构思和假设缺陷原因
      │
      ▼
运行数据和调试代码 ──► 未找到原因 ──► 返回构思假设
      │
      ▼
找到错误位置
      │
      ▼
定位和修复缺陷 ──► 修复缺陷 ──► 回归测试

11. 解决编程和调试问题

编程和调试需要开放的知识（设计语言、调试技术、设计模型等），实践中常遇到难以定位的错误。除了独立解决和团队帮助外，还可以利用 Stack Overflow、CSDN 等技术问答社区获取群体智慧。

12. 编写代码的输出

  • 源程序代码；
  • 程序单元测试报告（记录单元测试发现的缺陷）。

二、相关真题与解析

题目1（来源：2019年简答题第7题）

原题： 简述三种软件设计方法及其特点。

解析/答案：

课程中重点讲授了结构化软件设计方法和面向对象软件设计方法。结合软件工程知识体系，常见的三种设计方法可概括为：

  1. 结构化软件设计方法

    • 基本思想：以数据流图为主要输入，将系统分解为若干功能模块，形成层次化的模块结构；
    • 主要技术：面向数据流的变换型映射、事务型映射；
    • 特点：
    • 以功能模块为核心，强调自顶向下、逐步求精；
    • 输出为层次化的软件体系结构图；
    • 技术成熟，适合过程性强、需求相对稳定的系统；
    • 对需求变化适应性较弱。

  2. 面向对象软件设计方法

    • 基本思想：以面向对象需求模型（用例图、交互图、分析类图）为基础，对分析模型进行精化，得到子系统、构件、设计类等；
    • 主要技术：UML 建模（包图、类图、顺序图、状态图、活动图等）、设计模式、接口/继承/多态等机制；
    • 特点：
    • 抽象层次高，贴近现实世界；
    • 系统化支持体系结构、界面、数据、构件、子系统、用例、类设计等；
    • 易于扩展和变更，支持重用；
    • 适合大型复杂软件系统。

  3. 基于构件/重用的软件设计方法（课程作为补充思想）

    • 基本思想：尽可能重用已有的软件资产（构件、开源软件、框架等），通过接口组装成新系统；
    • 特点：
    • 以构件为组装单元，强调可独立部署和运行；
    • 显著提高开发效率和软件质量；
    • 需要良好的接口定义和构件管理机制。

题目3（来源：2020年回答题第4题）

原题： 模块独立性的两个度量概念，并指出设计原则及其内容。

解析/答案：

模块独立性的两个度量概念是内聚度（Cohesion）和耦合度（Coupling）。

1. 内聚度

内聚度衡量一个模块内部各成分之间彼此结合的紧密程度。内聚度越高，模块内部越“专一”。

类型 含义
偶然性内聚 模块内各成分关系松散，仅为完成一组功能而结合
逻辑性内聚 模块完成的诸任务在逻辑上相关
时间性内聚 模块内诸任务必须在同一时间段内执行
过程性内聚 模块内各成分相关且必须按特定次序执行
通讯性内聚 模块内各成分对数据结构的同一区域操作
顺序性内聚 模块内各成分与同一功能相关且顺序执行
功能性内聚 模块内各成分是一个整体，完成单个功能（最理想的内聚）

2. 耦合度

耦合度衡量模块之间的相关程度。耦合度越低，模块之间越独立。

类型 含义
非直接耦合 两个模块都不依赖对方而独立存在
数据耦合 两个模块通过参数交换信息，且仅限于数据
控制耦合 两个模块通过参数交换的信息包含控制信息
特征耦合 介于数据耦合和控制耦合之间
外部耦合 两个模块与同一外部环境相关联，如共享文件
公共耦合 模块间通过全局数据环境相互作用
内容耦合 一个模块直接使用另一模块内部数据或控制信息，或转入另一模块执行（应尽量避免）

3. 设计原则

高内聚、低耦合。

  • 模块内部应尽可能高内聚，使每个模块职责单一、功能完整；
  • 模块之间应尽可能低耦合，仅通过清晰接口交换必要信息；
  • 这样可降低系统复杂度、提高可维护性、可测试性和可重用性，也便于并行开发和后期修改。

题目4（来源：2019年简答题第4题）

原题： 面向对象设计6大原则。

解析/答案：

面向对象设计中常被提及的六大原则是 SOLID 原则加上迪米特法则（Law of Demeter）：

  1. 单一职责原则（Single Responsibility Principle, SRP）

    • 一个类只应有一个引起它变化的原因；
    • 即一个类只负责一项职责，避免职责过多导致耦合复杂。

  2. 开闭原则（Open/Closed Principle, OCP）

    • 软件实体应对扩展开放，对修改关闭；
    • 通过抽象、接口、继承等方式实现功能的扩展，而不修改已有代码。

  3. 里氏替换原则（Liskov Substitution Principle, LSP）

    • 子类应当能够替换父类并出现在父类能出现的任何地方；
    • 子类不应改变父类已有的行为约定。

  4. 依赖倒置原则（Dependency Inversion Principle, DIP）

    • 高层模块不应依赖低层模块，二者都应依赖抽象；
    • 抽象不应依赖细节，细节应依赖抽象。

  5. 接口隔离原则（Interface Segregation Principle, ISP）

    • 客户端不应被迫依赖它们不用的接口；
    • 应将庞大接口拆分为多个小而专的接口。

  6. 迪米特法则 / 最少知识原则（Law of Demeter, LoD）

    • 一个对象应尽可能少地了解其他对象；
    • 只与直接朋友通信，降低类之间的耦合。

这些原则与课程中讲授的“模块化、高内聚、低耦合、信息隐藏、关注点分离、软件重用”等软件设计基本原则相辅相成。

（3）协作图与顺序图的区别

  • 顺序图（Sequence Diagram）：
  • 强调消息传递的时间顺序；
  • 纵向为时间轴，横向为不同对象；
  • 通过对象生命线、活跃期（矩形条）清晰展示消息发生的先后和持续时间；

  • 适合分析用例实现的时序过程。

  • 协作图 / 通信图（Communication Diagram）：

  • 强调对象之间的组织关系和协作结构；
  • 对象之间的连接用连线表示，消息用带序号的箭头标注；
  • 通过编号表达消息顺序，但不像顺序图那样以时间为纵轴；
  • 更适合展示对象网络结构和消息在结构中的流动。

两者表达能力相同，可相互转换，只是侧重点不同。

本篇复习文档综合了第 3.1、4.1、5.1 讲全部知识点，并结合 2019、2020、2022 上期末考试真题进行了详细解析。建议在阅读时对照课件中的各类示意图（数据流图、结构图、UML 图）进行手绘练习，以加深对“从需求到设计再到实现”整体流程的理解。
