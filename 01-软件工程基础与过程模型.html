第1-3讲：软件工程基础与过程模型

一、知识点讲解

1.1 软件及其特点

本章知识图谱

1.1.1 从程序到软件：工程化开发的启发

早期人们编写程序往往是一步到位的个人创作，例如火车票购买、酒店预订、网上购物等应用，试图"通过一个步骤就将程序写出来"是不现实的。工程实践启示我们：建造狗窝与建造大厦有本质区别——前者简单、质量要求不高、一个人即可完成；后者复杂、质量要求很高、需要团队合作。同样，软件开发也需要经历多个相互关联、循序渐进的步骤，包括了解需求、设计图纸、进场施工等阶段，逐步产出需求分析、系统设计、程序代码以及配套文档和数据。

1.1.2 软件的定义

软件（Software） 是指在计算机系统的支持下，能够完成特定功能与性能的程序、数据和相关文档的集合。用公式表示：

$$ \text{软件} = \text{程序} + \text{数据} + \text{文档} $$

这些组成部分统称为软件制品（Software Artifact）：

  • 程序（代码）：由计算机执行的指令序列，是软件的可执行部分。
  • 数据：程序的加工处理对象和结果。包括用户数据、订单数据、交易数据、日志数据等。开发中需要考虑需要处理哪些数据、如何获取、表示、存储、检索和传输这些数据。
  • 文档：记录软件开发活动、阶段性成果、软件配置及变更的阐述性资料。包括软件需求文档、软件设计文档、软件测试文档、软件用户手册等。文档用于定义和理解软件、记录开发成果、辅助人员交流和促进软件管理。

从开发者的角度看，软件包括可执行部分（程序、部分数据）和不可执行部分（文档、数据）。

1.1.3 为什么需要文档

软件开发的涉及面广、参与人员多、产生的制品多，仅靠人脑记忆存在明显不足：

  • 任何事都记在脑子里，记不住；
  • 任何事都放在脑子里，理不清；
  • 任何事都存在脑子里，讲不明。

编写文档的目的包括：

  1. 阐述清楚：使内容具有逻辑性和条理性；
  2. 发现问题：通过文档化检查完整性、一致性和是否存在矛盾；
  3. 开展交流：便于不同背景人员（用户、开发人员、测试人员、管理人员）之间的沟通；
  4. 促进管理：管理软件开发成果和变更。

我国已发布《软件工程国标文档集合》，如 GB8567-88 和 GB8567-2006，规范了软件开发过程中应产生的文档。

1.1.4 软件概念带来的启示

软件 ≠ 程序，开发软件 ≠ 编写程序：

  • 软件涉及更多制品（文档、数据、代码），这些制品之间存在关联、相互依赖；
  • 软件开发需要做更多工作，对软件提出更高要求；
  • 开发具有一定规模和复杂性的软件系统与编写简单程序完全不同，必须采用工程方法。

1.1.5 软件生命周期

万物均有生命周期，软件也不例外。软件生命周期（Software Lifecycle） 是指软件从提出开发开始，到开发出系统、运行维护，直至最终退役所经历的全过程。生命周期内不同阶段具有不同特征，会产生不同的软件制品。

软件生命周期涵盖以下核心问题：

  • What：软件需求是什么？
  • How：如何构造出软件？
  • 部署与运行：编写的程序是否有缺陷？软件如何在实际环境中运行？

1.1.6 软件的特点

软件作为特殊的逻辑产品，具有以下显著特点：

  1. 逻辑性

    • 软件是逻辑产品，是思维活动的结果，而非物理活动的产物；
    • 软件不会磨损和老化，但可能因需求变化、环境变化而过时。

  2. 设计开发性

    • 软件是设计开发而成的，不是生产制造而成的；
    • 软件开发是智力密集型活动，强调创造性和设计能力。

  3. 易变性

    • 软件需求经常变化且难以把控；
    • 需求变化会影响软件的制品和开发过程本身。

  4. 复杂性

    • 规模大：代码行、模块数、接入人员、进程、数据量等都非常大；
    • 运行复杂：状态难以追踪和复现。

  5. 缺陷的隐蔽性

    • 缺陷隐藏在逻辑代码中，不像硬件那样直观显现；
    • 缺陷难以被发现和排除，可能潜伏很长时间。

1.1.7 军用软件的特殊性

军用软件除具有一般软件的特点外，还具有以下特殊性：

  • 运行环境更复杂和多样；
  • 与物理系统（如飞机、导弹）紧密联系在一起；
  • 对实时性和质量（尤其是可靠性）提出更高要求；
  • 对自身防护能力（如安全性）提出更高要求；
  • 需具备更强的灵活性、适应性和健壮性。

典型军用软件包括：导弹中的飞行控制软件、指挥信息系统、后勤保障软件等。

1.1.8 软件的分类

按照服务对象、功能和作用，软件可分为三类：

类别 服务对象 软件的功能 发挥的作用
应用软件 行业和领域的用户 为特定行业和领域问题提供基于软件的解决方案，创新应用领域的问题解决模式 提供更为便捷、快速、高效的服务
系统软件 各类应用软件 为应用软件运行和维护提供基础设施和服务，如加载、通讯、互操作、管理等 作为应用软件的运行环境
支撑软件 软件开发者和维护者 为软件系统的开发和维护提供自动和半自动的支持 提高软件开发效率和质量
  • 应用软件：面向特定应用领域的专用软件，如淘宝、12306、携程、微信、QQ 等。
  • 系统软件：对计算机资源进行管理，为应用软件的运行提供基础设施和服务，如操作系统、数据库管理系统、编译软件、软件中间件等。
  • 支撑软件：辅助软件开发和运维，帮助开发人员完成软件开发和维护工作，如 IDEA、Visual Studio、Eclipse 等。

1.1.9 开源软件

1.1.9.1 闭源软件

闭源软件是指软件代码不对用户开放的一类软件。用户购买时只获得可运行软件或服务，无法获得源代码，通常以许可证（License）方式授权用户使用。

闭源软件带来的问题：

  • 无法掌握软件内部实现情况（如是否存在安全漏洞和恶意代码）；
  • 无法对软件进行修改和完善；
  • 极大影响开发者的创新自由。

典型闭源软件和企业：微软的 Windows、Office，以及微软、IBM、Oracle 等。

1.1.9.2 开源软件的概念与优势

开源软件（Open Source Software） 是一种源代码可以自由获取和传播的计算机软件。其拥有者通过开源许可证赋予被许可人对软件进行使用、修改和传播的权利。

开源软件的好处：

  • 源代码可自由传播；
  • 激发创作者的热情；
  • 免费使用，降低成本。

典型开源软件：Linux、Ubuntu、Android AOSP、Chromium、VirtualBox、Nginx、Apache、Tomcat、PostgreSQL、MySQL、Kafka、RabbitMQ、TensorFlow、PyTorch、OpenClaw、Qwen、llama.cpp / Ollama 等。

开源软件正在逐步替代闭源软件：

  • 服务器操作系统领域：以 Linux、FreeBSD 为内核的操作系统逐步替代 Unix；
  • 桌面操作系统领域：以 Linux 为核心的开源操作系统正替代 Windows；
  • 数据库领域：PostgreSQL 等开源软件逐步替代 Oracle；
  • 浏览器领域：Chrome 及基于 Chromium 的浏览器市场占有率高。
1.1.9.3 开源软件的推动者
  1. 政府组织：我国政府将开源列入"十四五"国民经济和社会发展五年规划纲要。
  2. 开源组织：Apache 基金会、Linux 基金会、Eclipse 基金会、Open Source Initiative 等。
  3. 企业：Google、Microsoft、IBM、Oracle 等国外企业，以及华为、腾讯、阿里巴巴、百度、字节跳动、浪潮、联想等国内企业。
  4. 个人开发者：如 Linus Torvalds，以及 Github 上超过 6500 万程序员。
1.1.9.4 开源软件托管平台与社区
  • GitHub：全球最有影响力的开源软件托管平台，网址 www.github.com。
  • SourceForge：有影响力的开源软件开发平台和仓库，提供存储、协作和发布支持，网址 sourceforge.net。
  • Gitee：中国的开源软件托管平台，汇聚国内众多开发者和主要软件企业的开源项目，如华为 OpenHarmony、百度 PaddlePaddle 等，网址 www.gitee.com。
  • Stack Overflow：全球最大的软件开发知识分享平台，支持开发者交流讨论、分享经验和知识，促进开源软件开发。
1.1.9.5 开源软件的优势
  1. 采购和开发成本更低：开源软件通常免费，即使付费费用也较低廉。
  2. 软件质量更高、更安全：核心代码在公众视野中，代码问题（缺陷、安全漏洞）容易被人发现。
  3. 软件研制和交付更快：基于开源软件的项目开发可以更快速地交付产品。
  4. 软件功能更为强大：大量开发者参与软件开发、贡献代码、提出需求，不断完善功能。
1.1.9.6 开源软件许可证

开源许可证声明获得开源代码后拥有的权利，界定对开源作品可进行的操作以及禁止的操作，规范开源软件的使用要求和约束。

需要开源许可证的原因：

  • 在开源软件的自由创新与创业利益之间达成平衡；
  • 既支持开发者基于开源软件创新，也保护贡献者和创新者的利益；
  • 寻求商业运作模式，促进开源软件长期、持续、良性发展。

开源许可证主要分为两类：

  1. 宽松式开源许可证

    • 对用户限制很少，用户甚至可以将修改后的代码闭源；
    • 特点：代码使用无限制、用户自担代码质量风险、须披露原始作者；
    • 典型：BSD、Apache、MIT 许可证。

  2. Copyleft 式开源许可证

    • "Copyleft"是 Copyright 的反义词，意指可不经许可随意复制；
    • 更多限制，如分发二进制代码时须提供源代码，修改后的软件须保持相同许可证；
    • 典型：GPL、MPL 许可证。

Apache 许可证：

  • 该软件及其衍生品须继续使用 Apache 许可协议；
  • 修改源代码需在文档中声明；
  • 若软件基于他人源代码编写，需保留原始代码的协议、商标、专利声明及原作者声明；
  • 再发布软件中有声明文件时，需标注 Apache 许可协议及其他许可协议；
  • 典型软件：Hadoop、Apache HTTP Server、MongoDB。

GPL 许可证：

  • 自由复制，对复制数量和去处不做限制；
  • 自由传播，允许以各种形式传播；
  • 收费传播，允许出售，但必须让买家知道可免费获得；
  • 修改自由，允许增加或删除功能，但修改后的软件必须仍采用 GPL 许可证；
  • 典型软件：Linux。

不同许可证的差别可概括为：修改源码后是否可闭源、新增源码是否需遵循同样许可、是否需版权说明、是否可用开源软件名字促销衍生代码、是否需要对源代码修改提供说明等。总体上，MIT、BSD、Apache 约束较弱，LGPL、MPL、GPL 约束较强。

1.1.9.7 开源软件的利用
  1. 学习开源软件：学习核心技术、功能创意、架构设计、编码风格、模块封装等。
  2. 重用开源代码：将开源软件作为可重用软件资源库，实现更大粒度的软件重用。
  3. 参与开源建设：反馈缺陷、提出需求、讨论问题解决方案、贡献代码等，掌握开源技术、理解开源文化。

典型案例：Instagram 仅用 5 名软件工程师、十多款开源软件、8 周时间打造出最初版本，说明重用和集成开源软件是快速搭建高质量系统、提高开发效率和质量的有效手段。

据统计，78% 的公司基于开源运行，少于 3% 的公司完全不使用开源；89% 的公司认为利用开源大幅度提高了软件创新速度；64% 的公司参与开源，超过 66% 的公司优先考虑利用开源。贡献开源和利用开源已成为工业界的广泛共识和重要软件开发模式。

1.1.10 软件质量

软件质量是指软件满足给定需求的程度，是产品的生命线。

软件质量要素包括：

  1. 正确性（Correctness）：软件满足规格说明和用户要求的程度，即在预定环境下正确完成预期功能和非功能需求的能力。
  2. 可靠性（Reliability）：在规定条件下、限定时间范围内，软件系统完成预期功能、不引起系统故障的能力。
  3. 健壮性（Robustness）：在计算环境发生故障、输入无效数据或操作错误等意外情况下，软件仍能做出适当响应的程度。
  4. 有效性（Efficiency）：软件利用计算资源和存储资源以实现其功能的能力。
  5. 安全性：包括系统安全（Safety）和信息安全（Security）。系统安全是指软件能及时有效地避免给人员、设施、环境、经济等造成损害；信息安全是指软件能有效防控各类非法获取、传播和使用。
  6. 可维护性（Maintainability）：是否易于修改以更正错误、增强功能、适应新运行环境。
  7. 可移植性（Portability）：把软件从一种运行环境转移到另一种环境下运行的难易程度。
  8. 可重用性（Reusability）：软件模块、构件、设计方案等在其他软件开发中被再次使用的程度。
  9. 可理解性（Comprehensibility）：软件开发者或用户理解该软件系统的容易程度。
  10. 可信性（Dependability）：采取有效措施确认软件满足人们的要求和期望。
  11. 持续性（Sustainability）：面对各种突发异常事件，仍能提供令人满意服务的能力。
  12. 可用性（Usability）：使用和操作软件系统的难易程度。
  13. 互操作性（Interoperability）：软件系统与其他系统进行信息交换、协同工作的能力。

华为公司 2019 年第一号文件《全面提升软件工程能力与实践，打造可信的高质量产品》中提出把可信作为第一优先级，放在功能、特性和进度之上。可信软件的基本要求包括：

  • 安全性：抗击攻击，保护业务和数据的机密性、完整性和可用性；
  • 韧性（Resilience）：系统受攻击时仍能保持良好的运行状态，具有快速恢复能力；
  • 隐私性（Privacy）：保护用户数据及其隐私；
  • 可靠性和可用性：确保软件产品长期无故障运行，具备快速恢复和自我管理能力，提供可预期的、一致的服务。

1.1.11 当前软件特征的变化

1.1.11.1 软件的地位和作用
  • 软件深入到社会、经济、生活的方方面面，无处不在；
  • 作为创新工具，深刻改变各行各业，创新问题解决模式和方式，影响人们日常学习、生活和工作（如 12306 改变购票方式，淘宝和京东改变购物方式）；
  • 软件是实现行业和领域信息化融合与改造、实现创新性发展的使能技术和重要利器（如企业信息化改造、图书借阅系统等）；
  • 软件已成为人类社会的关键性基础设施，像电力、交通设施一样发挥基础性作用。
1.1.11.2 软件的运行环境

软件运行环境经历了从孤立、独立、局域和可控到分布、开放、动态、难控、无处不在计算环境的演变：

  • 大中小型机计算平台；
  • 基于 PC 的计算平台；
  • 基于局域网的分布计算平台；
  • 基于互联网的计算平台；
  • 无处不在的计算（从前端 PC 终端、可穿戴设备、智能手机到后端云中心、高性能计算中心）。
1.1.11.3 软件的形态

当前软件形态发生了根本性变化：

  1. 系统之系统：不是单一系统，而是诸多系统联盟和组合而成。
  2. 社会技术系统：不仅是社会技术系统，还是一种生态系统。社会、信息、物理等要素共存。
  3. 异构、多样的系统联盟：要素异构，客观存在且必然。
  4. 开放环境：不在封闭环境而在开放环境中，环境和系统相互作用，如互联网、战场环境。
  5. 动态适应系统：不是封闭系统，而是动态适应系统，要素、关系、联盟持续变化，边界不明确。

社会技术系统：由人、社会组织、物理设备、过程等要素共同组成和相互作用的人机物共生系统。软件系统无法单独存在，需与物理、社会系统交互作用。

系统之系统（System of Systems）：由一组面向任务、服务于不同对象的子系统构成。每个子系统可独立运作并提供相对独立功能，整个系统通过各独立系统间交互实现全局任务。

分布式异构系统：拥有大量形式多样、地理或逻辑上分布、部署在互联网上的软件实体。分布性是必须的，因为越来越多的应用本身就是分布的，且分布有助于提高可靠性和安全性。软件实体通常是异构的，异构性是必然。

动态演化系统：系统边界和需求具有不确定性和持续演变性。软件系统需要根据外部环境变化不断调整自身，包括体系结构和交互协作等，运维和运行交织在一起。例如城市交通管理系统、银行服务信息系统、医疗保证信息系统、作战指挥控制系统等。

系统联盟：大规模复杂信息技术系统由大量相对独立、自我控制和管理的系统组装而成。这些系统分属不同组织管理，共同工作和相互共存，并非一开始就设计好，而是演化而成，联盟动态变化。开发系统联盟需要采用社会技术观点，借助系统工程方法。

生态系统：

  • 运行生态：软件运行需要依靠生态支撑，从计算和物理设备、社会系统到 OS 和运行环境。
  • 演化生态：软件演化基于特定生态支撑，从最初的"源头"到最新"版本"。
  • 人员生态：涉及大量、开放的介入人员，包括使用者、开发者、运维者等。

生态的特点：共同环境 + 诸多要素 + 独立演化 + 相互依存。

1.1.11.4 软件系统的规模

构成软件系统的代码行数量、运行时的进程和线程及交互数量、处理的数据量、连接的设备和人员数量等不断增加：

  • Windows XP：约 3500 万行代码；
  • Windows Vista：约 5000 万行代码；
  • Windows 7：约 7000 万行代码；
  • BMW 7 系：软件总量超过 2 亿行；
  • 特斯拉 S：内嵌软件代码行总量超过 4 亿行；
  • 空客 A380：软件代码行总量超过 10 亿行。
1.1.11.5 软件系统的应用领域

软件系统广泛应用于：高性能计算、信息物理系统、智能机器人、云计算、健康医疗、城市交通、军事信息系统、航空航天等领域。

1.1.11.6 我国软件建设的挑战和使命

现实挑战：

  • 受制于人：微软 Windows 垄断桌面 OS，Google 垄断智能终端 OS；
  • 工业软件和科学计算软件等领域差距更大；
  • 整体实力不足，缺乏在国际上有影响力的软件企业和产品。

使命：加大、加快国产化软件系统的研发和应用，使命伟大、责任重大、任重道远。

我国通过工业 2025 等计划，逐步在关键领域成为世界第一阵营：AI（DeepSeek、Qwen、Kimi 等）、机器人（Unitree 宇树等）、无人机（DJI）、航天（天宫、神舟、嫦娥）、航空（第六代先进战机、Y20 等）等。

1.1.12 本讲小结

  • 软件由程序 + 文档 + 数据组成；
  • 软件特点：逻辑产品、设计开发、需求易变、系统复杂、缺陷隐蔽；
  • 开源软件：代码可自由获取和传播，需遵循许可证，充分利用开源软件；
  • 软件质量：多要素，如正确性、可靠性、可信性等；
  • 软件发生的变化：地位和作用、形态和复杂性、运行环境、系统规模。

1.2 软件工程概述

本章知识图谱

1.2.1 软件工程产生的背景

1.2.1.1 1950s-1960s 的计算机软件应用

20 世纪 50-60 年代，计算机软件应用领域发生变化：

  • 最早满足军方应用，如科学计算；
  • 逐步走向商业应用等新领域，如银行、航空等领域的事务处理；
  • 应用数量不断增长，计算机软件需求量上升；
  • 应用复杂性增加，用户和需求多样化。

典型案例：IBM OS/360 超大型软件项目（1960s 初）：

  • 复杂软件，支持多道程序，最多可同时运行 15 道程序；
  • 软件工程师超过 2000 人，花费超过 5 亿美元，工作量超过 5000 人年；
  • 被称为"有史以来最可怕的软件开发泥潭"；
  • Brooks 据此写出《人月神话》（The Mythical Man-Month），并获得图灵奖。
1.2.1.2 1960s 的个体作坊式软件开发

这一时期软件开发呈现"作坊式个体创作"特点：

  • 依靠个人能力，相互之间缺乏合作；
  • 关注计算存储时空利用，精雕细琢；
  • 程序规模小且功能单一；
  • 无系统性方法和标准流程。
1.2.1.3 个体作坊式创作带来的问题

作坊式的个体编程开发无法适应大批量和大规模软件系统的开发，主要表现在：

  • 开发过程：缺乏明确的步骤来开发软件系统；
  • 开发方法：缺乏系统方法来指导各项软件开发活动；
  • 开发管理：无法有效组织开发人员和管理软件产品；
  • 质量保证：无法保证软件开发活动和制品的质量。
1.2.1.4 软件危机的出现

随着软件规模扩大、需求增长、成功概率下降、软件生产率跟不上代码规模增长，软件开发面临严峻挑战。软件危机由此出现，主要表现包括：

  • 开发成本高：软件成本高，软硬件投资比发生急剧变化。例如美国空军 1955 年软件占计算机系统总费用的 18%，1970 年为 60%，1985 年达到 85%。IBM 360 OS 耗费 5000+ 人年，耗时 4 年（1963-1966），花费 2 亿多美元。
  • 进度难以控制：项目延期比比皆是，因进度问题取消的软件项目较常见，只有一小部分项目能够按期完成。
  • 质量难以保证：人总是会犯错误，软件开发的错误表现为多种形式：没有按需求开发、代码功能错误、性能不达标、交互界面用户不喜欢等。有些错误可能是致命的，如波音 737 MAX 事件，以及某教练飞机参数屏显范围 ±100 被重用于新战斗机（应为 ±300）导致的严重问题。
  • 软件维护困难：
  • 理解：读懂程序困难，尤其是他人程序；
  • 修改：程序脆弱，牵一发而动全身；
  • 出错：修改后易引入新错误；
  • 发现：有了错误后难以发现。
  • 失败风险很大：软件开发成功比例低，失败投资巨大。
1.2.1.5 软件危机的产生根源

软件危机的产生根源主要有三方面：

  1. 对软件复杂性和特殊性认识不清：软件是新生事物，对其特点、规律性和复杂性认识不够。
  2. 没有找到支持软件系统开发的有效方法：缺乏基础理论、关键技术、开发过程、支撑工具等。
  3. 缺乏成功软件开发实践及相应经验：需要系统总结、认真分析、充分借鉴、吸取教训。

因此，软件开发迫切需要理论和方法指导，软件工程应运而生。

1.2.2 软件工程的诞生

诞生标志：1968 年，在西德南部小城，NATO 科技委出资召开会议，11 个国家 50 位代表参加，主题是"如何解决软件危机"，会议提出了"软件工程"的概念和思想，标志着软件工程的诞生。

软件工程产生的动机：解决软件危机，使软件系统开发能够：

  • 快速；
  • 高效；
  • 低成本；
  • 高质量。

1.2.3 软件工程的定义

IEEE 93 定义：软件工程是将系统的、规范的、可量化的方法应用于软件的开发、运行和维护的过程，以及上述方法的研究。

具体内涵：

  • 系统化：提供完整和全面的解决方法，包括目标、原则、过程模型、开发活动、开发方法和技术等；
  • 规范化：支持各类软件系统的开发，包括语言标准、质量标准、编程标准、方法标准、能力及其改进标准等；
  • 可量化：工作量、成本、进度、质量等要素可以量化。

软件工程对软件开发的新认识：

  • 软件是产品（Product）：面向用户，存在质量、成本、利润等特征；
  • 软件开发是一项工程（Project）：存在约束，需要质量保证，进行组织管理；
  • 要按工程化方法来组织软件生产：分阶段分步骤实施，按计划开展开发活动，进行各种形式质量保证，采用行之有效的方法，借助各种工具支持。

软件开发方式的改变：从个体作坊式行为转向基于团队的协同开发方式，强调团队协作、分步实施、质量保证、开发技术和支持工具。

1.2.4 软件工程的三要素

软件工程包括三个核心要素：

  1. 过程（Process）

    • 从管理视角回答软件开发、运行和维护需要开展哪些工作、按照什么步骤和次序开展工作；
    • 对软件开发过程中涉及的人、制品、质量、成本、计划等进行有效和可量化的管理；
    • 典型成果：过程模型（瀑布模型、增量模型、原型模型、迭代模型、螺旋模型等）、方法（敏捷开发、群体化开发、DevOps）、管理（配置管理、质量管理、团队组织等）。

  2. 方法学（Methodology）

    • 从技术视角回答软件开发、运行和维护"如何做"的问题；
    • 为软件开发过程中的各项开发和维护活动提供系统性、规范性的技术支持；
    • 涉及如何理解和认识软件模型、如何用不同抽象层次的模型描述软件制品、采用什么建模语言描述软件模型等；
    • 典型成果：结构化软件开发方法学、面向对象软件开发方法学、基于构件的软件开发方法学。

  3. 工具（Tool）

    • 从工具辅助视角回答如何借助工具辅助软件开发、运行和维护；
    • 帮助软件开发人员更高效地运用软件开发方法学完成各项工作，提高效率、质量和交付进度；
    • 覆盖需求分析、软件设计、编码实现、软件测试、部署运行、软件维护、项目管理、质量保证等；
    • 典型成果：Eclipse、Visual Studio 等。

1.2.5 计算机辅助软件工程（CASE）

计算机辅助软件工程（Computer-Aided Software Engineering, CASE） 是指在软件工程活动中，开发人员按照软件工程的方法和原则，借助于计算机及其软件的帮助来开发、维护和管理软件产品的过程。

需要 CASE 的原因：

  • 软件及其开发很复杂，需要简化开发；
  • 产品数量多难以管理，需要提高效率；
  • 人的因素决定易出错，需要提高质量。

正所谓"工欲善其事，必先利其器"。

CASE 工具和环境：

  • CASE 工具：支持 CASE 的软件工具，如编辑器、编译器等，具有单一性。
  • CASE 环境：将 CASE 工具按统一标准和接口组装起来，使工具间、人员间、各个过程间能方便交互的集成环境。如 Visual Studio 将编辑、编译、调试、界面设计、安装程序生成等集成在一起。

CASE 工具分类：

  1. 代码编写：编辑、编译、分析、查找、代码生成等；
  2. 项目管理：工作量和成本估算、制定和跟踪计划、配置和版本管理；
  3. 软件建模：需求建模、UML 建模、数据建模等；
  4. 软件测试：测试用例自动生成、代码测试、缺陷报告等；
  5. 软件运维：软件运行、管理和维护。

1.2.6 软件开发的本质

软件开发的本质是：

$$ \text{软件开发} = \text{软件创作} + \text{软件生产} $$

  • 软件创作：基于软件开发者的经验和技能，借助于智慧，进行自由创新，如软件设计、编码实现等。
  • 软件生产：基于工程化手段，遵循约束和规范开展软件生产，如遵循过程、按照标准、质量保证等。

1.2.7 软件工程的目标

在成本、进度等约束下，指导软件开发和运维，开发出满足用户要求的足够好的软件。

1.2.8 软件工程的原则

软件工程遵循以下基本原则：

  1. 抽象和建模

    • 抽象：将开发活动所关注的要素提取出来，不关心的要素扔掉，形成与该开发活动相关的软件要素；
    • 建模：基于特定抽象，借助建模语言（如数据流图、UML 等），建立基于这些抽象的软件模型，促进对软件系统的准确理解。

  2. 模块化

    • 将软件系统功能分解和实现为若干模块，每个模块具有独立功能，模块间通过接口调用和访问；
    • 模块内部高内聚，模块间松耦合。

  3. 软件重用

    • 在软件开发过程中尽可能利用已有软件资源和资产（函数库、类库、构件库、开源软件、代码片段等）实现软件系统；
    • 努力开发出可被再次重用的软件资源（函数、类、构件等）；
    • 有助于提高开发效率、降低成本、满足工程约束、获得高质量产品。

  4. 信息隐藏

    • 模块内部信息（内部语句、变量等）对外不可见或不可访问，模块间仅交换完成功能所必需的信息（接口）；
    • 模块设计只对外提供可见接口，不提供内部实现细节；
    • 可提升模块独立性、减少错误向外传播、支持模块并行开发。

  5. 关注点分离

    • 将若干性质不同的关注点分离开来，在不同开发活动中针对不同关注点，随后整合结果形成完整视图；
    • 软件系统具有多面性：结构特征（体系结构）、行为特征（动作及输出结果）、高层需求模型、低层实现模型等；
    • 使开发者聚焦于某个关注点以简化任务，通过整合多个视点获得清晰、系统、深入的认识。

  6. 分而治之

    • 对复杂软件系统进行分解，形成子系统；若子系统仍复杂则继续分解，直至易于处理；
    • 然后整合子系统的问题解决得到整个系统的问题解决；
    • 有助于简化复杂系统开发、降低复杂性、提高效率、确保质量。

  7. 双向追踪原则

    • 正向追踪：当某软件制品变化时，追踪这种变化会对哪些制品产生影响，指导相关开发和维护工作；
    • 反向追踪：追踪产生变化的来源，明确变化原因及合理性；
    • 有助于确保制品间一致性、发现无意义变化，并基于变化指导开发和维护，确保质量。

  8. 工具辅助

    • 利用软件工具辅助软件开发和维护是行之有效的方法；
    • 尽可能借助计算机工具降低开发和维护负担，提高效率和质量。

1.2.9 软件工程的发展历程

软件工程发展大致经历以下阶段：

20 世纪 50-60 年代

  • 软件系统较为简单，计算机软件与硬件结合紧密；
  • "精雕细琢"程序代码，充分利用计算资源；
  • 出现黑客文化，倡导自由；
  • 成功案例：IBM OS/360 研制并投入商业使用；
  • 出现高级程序设计语言：Fortran、COBOL、LISP、ALGOL 等；
  • 开发手段落后、效率低、质量无法保证，引发软件危机。

20 世纪 70 年代

  • 计算机主机计算能力提升；
  • 软件向商业应用拓展，处理繁杂事务流程；
  • 程序设计语言和程序设计方法学成为研究热点，出现 PASCAL、C、Prolog、ML 等高级语言；
  • 提出瀑布软件开发过程模型、结构化软件开发方法学；
  • 形式化方法研究活跃；
  • 研制支持结构化方法和形式化方法的 CASE 工具和环境；
  • 开始采用定量方法指导开发、管理和质量保证。

20 世纪 80 年代

  • 计算机软件应用领域和范围不断扩大，软件数量、系统规模和复杂性增长；
  • 产生面向对象程序设计技术，如 Smalltalk、C++；
  • 提出 SW-CMM（软件能力成熟度模型）；
  • 软件重用被视为解决软件危机的现实可行途径；
  • CASE 工具和环境研制使用成为热点；
  • 软件工程标准化工作活跃，成果丰硕。

20 世纪 90 年代

  • 局域计算环境流行，互联网应用开始出现；
  • 面向对象技术趋于成熟，面向对象分析和设计方法学研究活跃，形成系统化面向对象软件工程；
  • 制定 UML，产生统一软件开发过程 RUP；
  • 软构件技术快速发展，萌生软件体系结构和软件设计模式研究与实践；
  • 开源软件及技术开始出现；
  • 人机交互技术取得长足进步。

21 世纪前十年

  • 互联网技术日趋成熟，信息技术快速发展，软件数量不断增长，越来越多软件部署在互联网上运行并提供服务；
  • 产生网构软件技术、自适应软件工程、可信软件技术、面向主体软件工程等；
  • 群体化软件开发技术在开源软件开发实践中广泛应用；
  • 面向服务软件工程研究与实践；
  • 敏捷开发方法应用；
  • 模型驱动软件开发技术研究与应用；
  • 软件可信技术研究与实践活跃。

近十年

  • 移动互联网快速发展，应用软件需求激增，信息系统人机物融合趋势突出；
  • 人类进入"软件定义一切"时代，对软件开发和运维提出严峻挑战；
  • 越来越多企业和个人参与开源软件实践，形成规模庞大的开源软件生态；
  • DevOps 方法广泛应用；
  • 智能化软件工程（AI for SE）研究活跃。

1.2.10 我国软件工程发展

我国软件工程起步于 1980 年前后，过去四十多年取得长足进步：

  • 1980s：软件自动化开发和形式验证；
  • 1980s：软件开发方法学和 CASE 工具及集成环境；
  • 21 世纪初期：网构软件技术；
  • 21 世纪以来：可信软件技术；
  • 近 10 多年：开源软件研究与实践。

1.2.11 软件工程发展的时代特点

  • 与时代的计算技术和信息技术发展紧密联系；
  • 20 世纪 60 年代到 80 年代中后期：部署运行在单机或主机上；
  • 20 世纪 80 年代中后期到 90 年代：个人计算机和局域网出现；
  • 21 世纪：互联网应用普及，更多系统部署运行在动态、难控、不确定的互联网环境上；
  • 2010 年以来：泛在计算环境和移动互联网应用普及，信息系统人机物融合特征日益突出。

1.2.12 软件工程发展的技术特点

  1. 软件抽象层次越来越高：二进制编码 → 结构化编程 → 结构化开发方法学 → 面向对象编程 → 面向对象开发方法学 → ……
  2. 软件重用粒度越来越大：函数和过程 → 类 → 构件 → 开源软件 → ……
  3. 软件开发理念不断变化：
    • 以文档为中心与以代码为中心；
    • 从个体、团队到群体的开发组织；
    • 从还原论到演化论。

1.2.13 软件工程的多学科交叉

软件工程是一门交叉学科，与计算机科学与技术、数学、数据科学、复杂性科学、人工智能、工程和管理科学、认知和社会科学等相互交叉：

  • 复杂性科学帮助认识大型复杂软件系统、揭示内在机理、指导方法研究与实践；
  • 多学科交叉是软件工程的重要特征。

1.2.14 软件工程的变与不变

不变的：

  • 目标：解决软件危机；
  • 学科交叉；
  • 基本原则：软件重用、模块化、问题分解等。

变化的：

  • 对软件的理解和认识：社会技术视点；
  • 交叉更多的学科；
  • 方法和手段：敏捷方法、群体化开发、智能化软件工程等。

1.3 软件过程模型

本章知识图谱

1.3.1 软件开发的特点

软件开发具有以下特点：

  1. 基于智力的协作过程

    • 智力活动：基于逻辑思维构造软件；
    • 交流协作：软件工程师、用户间的交流和讨论。

  2. 软件项目内在复杂性

    • 介入的人多、考虑的内容多、产生的制品多；
    • 不同要素间存在关联。

  3. 循序渐进的开发过程

    • 开展有序的开发活动，如编码、分析、设计；
    • 体现工程思想：按步骤、分阶段。

1.3.2 软件过程

过程（Process） 包含三个要素：

  • 活动：明确要做哪些事情；
  • 关系：活动间存在逻辑关系，如依赖和先后次序。

软件过程（Software Process） 是指按照项目进度、成本和质量要求，遵循用户需求，开发和维护软件、管理软件项目的一系列有序软件开发活动。软件开发活动包括：

  • 技术活动：如需求分析、设计、编码、测试等；
  • 管理活动：如计划、跟踪、配置管理、质量管理等。

1.3.3 软件过程模型

软件过程模型（Software Process Model） 定义了软件开发的具体活动以及活动间的逻辑关系。它回答了"按照什么样的过程来有序地开发软件"这一问题。

软件过程模型涉及：

  • 投入：人员、工具、资源和成本等；
  • 活动：任务、目标、输入和输出；
  • 关系：活动间的关系和次序。

1.3.4 软件过程模型的产生背景

软件工程产生之前，软件开发是"作坊式个人创作"：

  • 聚焦于编写代码；
  • 依靠个体技能，缺乏合作；
  • 关注时空利用，精雕细琢；
  • 程序规模小且功能单一；
  • 无系统性方法和标准流程。

OS/360 等超大型项目的失败促使人们认识到需要系统、规范性的软件过程模型指导。

1.3.5 典型的软件过程模型

常见的软件过程模型包括：

  • 瀑布模型（Waterfall Model）
  • 增量模型（Incremental Model）
  • 迭代模型（Iterative Model）
  • 原型模型（Prototype Model）
  • 螺旋模型（Spiral Model）
  • 基于构件的过程模型
  • UP 模型（统一过程模型，Unified Process）：以用例驱动、以体系结构为中心、迭代和增量的软件开发过程

每种模型有其各自特点和适用场所。

1.3.6 瀑布模型

瀑布模型于 1970 年提出，是首个软件过程模型。

1.3.6.1 瀑布模型的阶段

瀑布模型将软件开发过程划分为若干顺序执行的阶段，典型阶段包括：

  1. 需求分析（Requirement Analysis）

    • 任务：定义软件需求，包括功能需求和非功能需求。
    • 关注点：要做什么？（What, Problem）问题是什么？
    • 层次和视角：用户角度，仅描述问题和需求。
    • 依据：用户的期望和要求。
    • 方法：与用户交流商讨，采用抽象、问题分解、多视点等技术。
    • 产出：软件需求模型、软件需求文档、软件确认测试计划等文档类制品。

  2. 概要设计（Architecture Design）

    • 任务：建立软件总体架构、制定集成测试计划。
    • 关注点：软件高层设计？（How, Solution）问题如何解决？
    • 层次和视角：宏观、全局、整体、战略性。
    • 依据：软件需求文档。
    • 方法：自顶向下、逐步求精、抽象、模块化、局部化、信息隐藏等。
    • 产出：软件概要设计模型、软件概要设计文档、软件集成测试计划等文档类制品。

  3. 详细设计（Detailed Design）

    • 任务：设计模块内部细节（算法、数据结构），制定单元测试计划。
    • 关注点：详细设计？（How, Solution）问题如何解决？
    • 层次和视角：微观、局部、细节性。
    • 依据：概要设计文档、软件需求文档。
    • 方法：高质量软件设计原则，如单入口单出口。
    • 产出：软件详细设计模型、软件详细设计文档、单元测试计划等文档类制品。

  4. 编码实现（Implementation）

    • 任务：编写程序代码并进行单元测试和调试。
    • 关注点：如何最终做出这个东西？（How, Code）实际解决问题。
    • 层次和视角：最终的实现代码。
    • 依据：软件概要设计文档、详细设计文档、单元测试计划。
    • 方法：采用某种程序设计语言（如 C、C++、Java）。
    • 产出：经过单元测试的源程序代码等程序类制品。

  5. 集成测试（Integration Test）

    • 任务：组装软件模块并进行测试以发现问题。
    • 关注点：集成后软件中的缺陷（Bug）。问题解决如何？软件有缺陷吗？
    • 层次和视角：自底向上组装、全局。
    • 依据：软件概要设计文档、软件集成测试计划。
    • 方法：软件集成测试工具。
    • 产出：经过集成测试、修复缺陷的源程序代码，集成测试报告等数据、文档和代码类制品。

  6. 确认测试（Validation Test）

    • 任务：测试软件是否满足用户需求。
    • 关注点：软件在满足用户需求方面是否存在缺陷。问题解决如何？软件有缺陷吗？
    • 层次和视角：从用户角度，聚焦需求是否得以正确实现。
    • 依据：软件确认测试计划、软件需求文档。
    • 方法：软件测试支撑工具。
    • 产出：经过确认测试、修复缺陷后的代码，软件确认测试报告等数据、文档和代码类制品。
1.3.6.2 瀑布模型的特点
  • 与软件生命周期相互一致；
  • 每个活动结束后需要评审；
  • 相邻活动间存在因果关系。
1.3.6.3 瀑布模型的优点
  • 简单，一目了然；
  • 易于理解、掌握、应用和管理；
  • 适合于需求易于定义、不易变动的软件系统。
1.3.6.4 瀑布模型的局限性
  • 需求确定，过于理想化；
  • 缺乏变通，难以应对变化；
  • 由于软件需求具有易变、多变的特点，瀑布模型难以适应实际需求变化。
1.3.6.5 改进的瀑布模型：带反馈和回溯

为应对瀑布模型的不足，可引入反馈和回溯机制：

  • 后期活动发现问题后，可返回到前面活动加以解决；
  • 提高了过程模型的灵活性。

但改进模型仍存在不足：软件开发处于动荡之中，需等到所有功能实现后才能得到可运行软件。

1.3.7 增量模型

增量模型（Incremental Model） 的核心思想是渐进式、增量式地实现软件功能。

1.3.7.1 增量模型的过程

需求分析和概要设计一次性完成，之后将软件划分为多个增量，每个增量经历详细设计、编码实现、集成测试、确认测试等阶段，逐步交付。

1.3.7.2 增量模型的优点
  • 渐进快速交付；
  • 可并行开发多个增量，提高效率。
1.3.7.3 增量模型的局限性
  • 软件需求可确定且不易于变化时才适用；
  • 若需求频繁变化，增量划分和开发计划难以稳定。

1.3.8 迭代模型

迭代模型（Iterative Model） 通过多次迭代逐步完善软件。

1.3.8.1 迭代模型的过程

每次迭代都经历一个完整的过程（需求分析、概要设计、详细设计、编码实现、软件测试），每次迭代完成部分可确定的软件需求。

1.3.8.2 迭代模型的特点
  • 每次迭代是一个完整过程；
  • 体现"小步快跑"的开发理念；
  • 适合需求难导出、不易确定且持续变动的软件。
1.3.8.3 迭代模型的局限性
  • 迭代多少次不确定；
  • 管理较为复杂。
1.3.8.4 增量模型与迭代模型的区别
维度 增量模型 迭代模型
核心思想 将软件功能分块，逐块增量实现 多次循环，每次完善部分需求
每次产出 一个可交付的、相对完整的功能增量 一个可运行的、逐步完善的版本
需求要求 需求整体可确定，变化不大 需求难以一次性确定，持续变化
适用场景 需求明确、可分解为模块的系统 需求模糊、需要不断探索的系统

简言之，增量模型是"一块一块地做"，迭代模型是"一遍一遍地改"。

1.3.9 原型模型

原型模型（Prototype Model） 是为解决需求获取难题而提出的模型。

1.3.9.1 软件需求获取的难题
  • 软件需求非常关键，是软件开发的基础和验收依据；
  • 用户往往讲不清楚需求有哪些、是什么（说不清、道不明），尤其当软件复杂庞大时；
  • 用户与软件工程师对需求的理解存在偏差，描述可能存在歧义、二义性、不准确等问题；
  • 如果需求分析结果不正确、不完整、有歧义，会导致后续开发工作出现严重偏差。
1.3.9.2 原型模型的过程
  1. 采集和细化需求；
  2. 快速设计；
  3. 建造原型（如用户界面、执行流程）；
  4. 客户使用评价原型；
  5. 获得反馈；
  6. 根据反馈继续开发软件系统。
1.3.9.3 原型模型的特点
  • 软件原型作为交流载体和媒介；
  • 支持用户参与到软件开发中；
  • 持续、渐进地导出用户要求。
1.3.9.4 原型模型的适用场景

适合于需求难导出、不易确定且持续变动的软件。

1.3.10 螺旋模型

螺旋模型（Spiral Model） 是一种风险驱动的过程模型。

1.3.10.1 软件风险

软件风险是指使软件开发受到影响和损失、甚至导致失败的、可能会发生的事件。

1.3.10.2 螺旋模型的特点
  • 集成迭代模型和原型模型；
  • 引入风险分析，是风险驱动的过程模型；
  • 每个迭代包含四个阶段和若干活动；
  • 适合于需求不明确、开发风险高、开发过程中需求变更大的软件项目。
1.3.10.3 螺旋模型的局限性
  • 管理复杂；
  • 对风险评估能力要求高。

1.3.11 不同软件过程模型的比较

1.3.12 软件过程模型的选择

1.3.12.1 选择依据

选择合适的软件过程模型需要综合考虑：

  1. 软件项目的特点

    • 业务领域是否明确；
    • 软件需求是否易于确定；
    • 用户需求是否会经常性变化；
    • 是否可以预估潜在软件开发风险。

  2. 软件开发团队的水平

    • 结合团队能力和水平选择过程模型，避免无法掌控和驾驭。

  3. 分析软件过程模型特点

    • 优缺点以及适合的场所。
1.3.12.2 选择示例
  • 互联网应用软件：需求不确定且快速变化，如 12306 APP、微信、淘宝。选用瀑布模型不合适，迭代模型较为合适。
  • 装备软件：需求确定且较为稳定，如飞行控制软件。可考虑选用瀑布模型，用迭代模型不是很合适。
1.3.12.3 课程实践项目的选择思考

课程实践项目通常具有"有创意、上规模"等特点（如代码量 > 15000+ LOC），团队多为学生团队，经验有限。选择过程模型时应结合项目特点和团队特点：

  • 需求可能不完全明确，存在探索性；
  • 团队需要循序渐进地学习和开发；
  • 迭代模型或增量模型可能较为合适，便于小步快跑、及时调整；
  • 也可结合原型模型辅助需求确认。

1.3.13 传统软件过程模型的特点和不足

传统软件过程模型（瀑布、增量、迭代、原型、螺旋等）存在以下共性问题：

  • 软件开发和运维的大量工作用于撰写软件文档，而非编写程序代码；
  • 软件开发过程中花费大量时间和精力用于文档评审，以确保软件质量；
  • 一旦软件需求发生变化，开发人员需要修改需求文档，并据此调整其他一系列文档，最后再修改程序代码；
  • 需要较长时间才能得到可运行软件系统。

这些模型属于以文档为中心的重型软件开发方法，非常笨重，难以快速应对需求变化。

1.3.14 敏捷软件开发方法

为应对传统重型方法的不足，敏捷软件开发方法（Agile Method） 应运而生：

  • 重视人和交互；
  • 重视可运行软件系统；
  • 重视客户合作；
  • 重视响应用户需求变化。

敏捷方法主张少写软件文档，以代码为中心，快速响应变化。这是传统过程模型之后的重要发展方向。

1.3.15 本讲小结

  • 软件开发需要过程指导，明确步骤、活动、次序和关系；
  • 存在多样化的软件过程模型：瀑布、增量、迭代、原型、螺旋等，各自有优缺点；
  • 选择合适的软件过程模型需考虑项目特点、团队经验和模型优缺点；
  • 传统软件过程模型以文档为中心，非常笨重，难以快速应对需求变化，因此催生了敏捷等轻型方法。

题目1（来源：2019年软件工程考试卷）

原题： 软件的定义；软件工程生命周期及各阶段工作。

解析/答案：

（1）软件的定义

软件是指在计算机系统的支持下，能够完成特定功能与性能的程序、数据和相关文档的集合。即：

$$ \text{软件} = \text{程序} + \text{数据} + \text{文档} $$

其中，程序是计算机可执行的指令序列；数据是程序的加工处理对象和结果；文档是记录软件开发活动、阶段性成果、软件配置及变更的阐述性资料。

（2）软件生命周期及各阶段工作

软件生命周期是指软件从提出开发开始，到开发出系统、运行维护，直至最终退役所经历的全过程。按照瀑布模型，典型阶段包括：

  1. 需求分析：定义软件的功能需求和非功能需求，产出软件需求模型、需求文档和确认测试计划。
  2. 概要设计：建立软件总体架构，制定集成测试计划，产出概要设计模型、概要设计文档和集成测试计划。
  3. 详细设计：设计模块内部细节（算法、数据结构），制定单元测试计划，产出详细设计模型、详细设计文档和单元测试计划。
  4. 编码实现：编写程序代码并进行单元测试和调试，产出经过单元测试的源程序代码。
  5. 集成测试：组装软件模块并进行测试，发现模块间接口缺陷，产出集成测试报告。
  6. 确认测试：从用户角度测试软件是否满足需求，产出确认测试报告。

题目2（来源：2019年、2020年软件工程考试卷）

原题： 什么是软件危机？软件危机的表现及原因。

解析/答案：

（1）软件危机的定义

软件危机是指在计算机软件开发和维护过程中遇到的一系列严重问题。它是随着软件规模扩大、复杂性增加而产生的，是软件开发从个体作坊式创作走向大规模团队开发时必然面临的挑战。

（2）软件危机的表现

  1. 开发成本高：软件成本在计算机系统总成本中的比例急剧上升。例如美国空军软件费用占比从 1955 年的 18% 上升到 1985 年的 85%。
  2. 进度难以控制：项目延期比比皆是，因进度问题取消的软件项目较常见，只有少量项目能按期完成。
  3. 质量难以保证：开发错误表现为未按需求开发、功能错误、性能不达标、用户界面不友好等，有些错误甚至可能是致命的。
  4. 软件维护困难：理解他人程序困难，程序脆弱易引入新错误，错误难以发现。
  5. 失败风险很大：软件开发成功比例低，失败投资巨大。

（3）软件危机产生的原因

  1. 对软件复杂性和特殊性认识不清：软件是新生事物，对其特点、规律性和复杂性认识不够。
  2. 没有找到支持软件系统开发的有效方法：缺乏基础理论、关键技术、开发过程、支撑工具等。
  3. 缺乏成功软件开发实践及相应经验：需要系统总结、认真分析、充分借鉴、吸取教训。

解决软件危机需要采用工程化方法，这正是软件工程产生的背景和动机。

题目3（来源：2020年软件工程考试卷）

原题： 生命周期分为哪几个阶段，每个阶段的内容是什么？

解析/答案：

软件生命周期是软件从提出开发到最终退役的全过程。以瀑布模型为例，主要分为以下阶段：

  1. 需求分析

    • 内容：定义软件的功能需求和非功能需求，明确"要做什么"。
    • 从用户角度出发，通过交流、抽象、问题分解、多视点等技术导出需求。
    • 产出：软件需求模型、软件需求文档、软件确认测试计划。

  2. 概要设计

    • 内容：建立软件总体架构，制定集成测试计划，解决"问题如何解决"的宏观问题。
    • 采用自顶向下、逐步求精、抽象、模块化、信息隐藏等方法。
    • 产出：软件概要设计模型、概要设计文档、集成测试计划。

  3. 详细设计

    • 内容：设计模块内部细节，包括算法、数据结构等，制定单元测试计划。
    • 从微观、局部、细节性视角开展设计。
    • 产出：详细设计模型、详细设计文档、单元测试计划。

  4. 编码实现

    • 内容：编写程序代码，进行单元测试和调试，最终产生可执行代码。
    • 依据概要设计和详细设计文档，采用某种程序设计语言实现。
    • 产出：经过单元测试的源程序代码。

  5. 集成测试

    • 内容：将模块组装起来进行测试，重点发现模块间接口的缺陷。
    • 依据概要设计文档和集成测试计划。
    • 产出：经过集成测试修复缺陷后的代码、集成测试报告。

  6. 确认测试

    • 内容：从用户角度验证软件是否满足需求，检查功能和性能缺陷。
    • 依据需求文档和确认测试计划。
    • 产出：经过确认测试修复缺陷后的代码、确认测试报告。

每个阶段结束后通常需要进行评审，相邻阶段之间存在因果关系，也可通过反馈和回溯机制提高灵活性。

题目4（来源：2022年软件工程试题A/B卷）

原题： 软件生命周期各阶段的任务是什么？（5分）

解析/答案：

软件生命周期各阶段的主要任务如下：

  1. 需求分析：明确软件需要做什么，定义功能需求和非功能需求，形成需求规格说明。
  2. 概要设计：建立软件总体结构和体系架构，确定模块划分及模块间接口，制定集成测试计划。
  3. 详细设计：对每个模块进行具体设计，包括算法、数据结构、接口细节等，制定单元测试计划。
  4. 编码实现：根据设计文档编写程序代码，进行单元测试，产出可运行的程序单元。
  5. 集成测试：将各模块按设计要求组装起来，测试模块间接口和协作，发现并修复缺陷。
  6. 确认测试：依据需求规格说明，从用户角度验证软件是否满足需求，确认软件可否交付使用。

以上阶段体现了"按步骤、分阶段"的工程化思想，每个阶段都有明确的输入、任务和输出。

题目5（来源：2022年软件工程试题A/B卷综合分析题）

原题： 软件系统建模是软件开发过程中的重要环节，通过构建软件系统模型可以帮助系统开发人员理解系统、抽取业务过程和管理系统的复杂性，也可以方便各类人员之间的交流。说明软件系统开发中常用的建模方法有哪几类？阐述每种方法的特点及其适用范围。

解析/答案：

软件系统建模方法主要分为以下几类：

（1）结构化建模方法

  • 特点：强调"自顶向下、逐步求精"，采用功能分解思想，使用数据流图（DFD）、数据字典、结构化语言等工具描述系统。
  • 适用范围：需求明确、以数据处理为核心的信息系统；适合中小型系统和传统事务处理系统。

（2）面向对象建模方法

  • 特点：以对象为基本单位，将数据和操作封装在一起，采用 UML 统一建模语言，从用例视角、行为视角、结构视角等多角度描述系统。
  • 适用范围：复杂、大型、需求易变的系统；当前软件开发的主流方法，广泛应用于企业信息系统、嵌入式系统等。

（3）基于构件的建模方法

  • 特点：将系统看作由若干可复用构件组装而成，强调构件的接口、复用和组装。
  • 适用范围：需要大量复用已有软件资产、快速构建系统的场景；适合分布式系统、企业应用集成等。

（4）形式化建模方法

  • 特点：采用严格的数学符号和规则描述系统，能够进行严格推理和验证，保证系统的正确性。
  • 适用范围：对安全性、可靠性要求极高的关键系统，如航空航天、轨道交通、核能控制等领域。

（5）原型建模方法

  • 特点：通过快速构建可运行的原型系统，与用户反复交流确认需求。
  • 适用范围：需求不明确、用户难以一次性说清需求的系统，常用于需求获取阶段。

不同建模方法各有优势，实际项目中常结合使用，以充分发挥各自长处。

题目6（综合理解题）

原题： 简述软件工程的定义、三要素及主要目标。

解析/答案：

（1）软件工程的定义

根据 IEEE 93 定义，软件工程是将系统的、规范的、可量化的方法应用于软件的开发、运行和维护的过程，以及上述方法的研究。

  • 系统化：提供完整和全面的解决方法；
  • 规范化：支持各类软件系统开发，遵循各种标准；
  • 可量化：工作量、成本、进度、质量等可以量化。

（2）软件工程的三要素

  1. 过程（Process）：从管理视角回答软件开发、运行和维护需要开展哪些工作、按什么步骤和次序开展。典型成果包括瀑布模型、增量模型、敏捷方法、DevOps、配置管理、质量管理等。
  2. 方法学（Methodology）：从技术视角回答"如何做"的问题，提供系统性、规范性的技术支持。典型成果包括结构化方法、面向对象方法、基于构件的方法等。
  3. 工具（Tool）：从工具辅助视角回答如何借助工具提高效率和质量。典型成果包括 Eclipse、Visual Studio 等 CASE 工具和环境。

（3）软件工程的主要目标

在成本、进度等约束条件下，指导软件开发和运维，开发出满足用户要求的"足够好"的软件。

题目7（综合理解题）

原题： 比较瀑布模型、增量模型、迭代模型、原型模型和螺旋模型的特点和适用场景。

解析/答案：

模型 核心思想 主要特点 适用场景 管理难度
瀑布模型 按阶段顺序推进 与软件生命周期一致，阶段间有因果关系，每阶段需评审 需求明确、变动小的项目 低
增量模型 分增量渐进交付 可快速交付部分功能，支持并行开发 需求较明确、可分解为模块的项目 中等
迭代模型 多次迭代完善 每次迭代是完整过程，体现"小步快跑" 需求不确定、持续变化的项目 中等
原型模型 以原型导出需求 通过原型与用户交流，持续确认需求 需求难以表述、获取困难的项目 较低
螺旋模型 风险驱动的迭代 集成迭代和原型模型，引入风险分析 风险高、需求变化大的大型项目 高

选择建议：

  • 需求明确、稳定：优先考虑瀑布模型或增量模型；
  • 需求模糊、易变：优先考虑迭代模型或原型模型；
  • 风险高、规模大：优先考虑螺旋模型；
  • 实际项目中，常根据项目特点和团队能力灵活组合使用。

题目8（来源：2022年软件工程试题A/B卷）

原题： 简要说明协作图在描述对象之间的动态交互关系时和顺序图存在哪些区别。

解析/答案：

顺序图和协作图（通信图）都是 UML 中用于描述对象间动态交互的图，二者表达能力相同，可以相互转换，但侧重点不同：

  1. 关注点不同

    • 顺序图：强调消息传递的时间顺序，纵向为时间轴，能清晰展示消息发生的先后次序和对象生命线。
    • 协作图：突出对象间的合作关系和链接结构，强调哪些对象之间发生了交互以及它们之间的结构关系。

  2. 布局方式不同

    • 顺序图：对象横向排列，时间纵向展开，消息沿纵向时间轴传递。
    • 协作图：对象之间的位置关系更为重要，消息标注在对象之间的链接上，用序号表示消息顺序。

  3. 适用场景不同

    • 顺序图：适合强调时间顺序、交互流程复杂的场景，如分析某个用例的实现过程。
    • 协作图：适合强调对象结构关系、展示哪些对象参与协作的场景。

简言之，顺序图强调"何时发生"，协作图强调"谁与谁交互"。

题目9（综合理解题）

原题： 简述开源软件的概念、优势及主要许可证类型。

解析/答案：

（1）开源软件的概念

开源软件是一种源代码可以自由获取和传播的计算机软件，其拥有者通过开源许可证赋予被许可人对软件进行使用、修改和传播的权利。但开发者虽然可自由获取源代码，在使用时仍需遵循相关开源协议。

（2）开源软件的优势

  1. 采购和开发成本更低：通常免费，即使付费也较为低廉；
  2. 软件质量更高、更安全：核心代码公开，缺陷和安全漏洞容易被人发现；
  3. 软件研制和交付更快：可基于开源软件快速构建和交付产品；
  4. 软件功能更为强大：大量开发者参与，不断完善功能和创新。

（3）主要许可证类型

  1. 宽松式许可证：对用户限制少，允许闭源修改后的代码，典型有 BSD、Apache、MIT 许可证。
  2. Copyleft 式许可证：限制较多，要求分发二进制时提供源代码，修改后保持相同许可证，典型有 GPL、MPL 许可证。

例如，Linux 采用 GPL 许可证，Hadoop、Apache HTTP Server、MongoDB 采用 Apache 许可证。

题目10（综合理解题）

原题： 简述软件工程的基本原则。

解析/答案：

软件工程的基本原则包括：

  1. 抽象和建模：抽象提取关键要素，忽略不相关细节；建模借助建模语言建立软件模型，促进准确理解。
  2. 模块化：将系统分解为若干功能独立的模块，模块内部高内聚、模块间松耦合，通过接口交互。
  3. 软件重用：尽可能利用已有软件资源（函数库、类库、构件、开源软件等）提高开发效率、降低成本。
  4. 信息隐藏：模块内部实现细节对外不可见，仅通过接口交互，提升独立性，减少错误传播。
  5. 关注点分离：将不同性质的关注点分离处理，再整合形成完整视图，简化开发任务。
  6. 分而治之：将复杂系统分解为子系统，继续分解直至易于处理，再整合得到整体解决方案。
  7. 双向追踪原则：正向追踪变化影响范围，反向追踪变化来源，确保制品一致性。
  8. 工具辅助：借助计算机工具辅助开发和维护，提高效率和质量，即"工欲善其事，必先利其器"。

这些原则贯穿于软件工程的各个方面，是指导软件开发活动的重要准则。
