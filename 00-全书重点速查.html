二、SE1 ~ SE13 考点速查

SE1 什么是软件

核心概念：

  • 软件 = 程序 + 数据 + 文档。
  • 软件是逻辑产品，非物理产品；设计开发而成，需求易变，系统复杂，缺陷隐蔽。
  • 软件特征的变化：规模、形态、环境、开发方式、价值、质量观等不断变化。
  • 开源软件：源代码公开、可自由使用/修改/分发，遵循开源许可证。
  • 软件质量：软件满足明确和隐含需求能力的特性总和。

题型：填空题常考定义、组成、特征；简答题可考「软件与硬件区别」「软件质量特征」。

SE2 什么是软件工程

核心概念：

  • 软件危机：软件开发与维护中成本高、进度失控、质量差、维护难等严重问题。
  • 1968 年 NATO 会议提出「软件工程」概念。
  • 软件工程三要素：过程、方法、工具。
  • 软件工程基本内涵：方法、过程、工具、原则、目标。
  • 核心原则：抽象、建模、模块化、信息隐藏、关注点分离、可追踪性、复用等。

题型：填空题考三要素、软件危机表现；简答题考「软件工程定义」「软件危机原因与表现」。

SE3.1 基本软件过程模型

五种基本模型：

模型 指导思想 特点 适用场景
瀑布模型 阶段顺序推进 文档驱动，阶段评审，回溯成本高 需求明确、变动小的项目
增量模型 分增量渐进交付 快速交付部分功能，可并行开发 需求较明确、可模块化
迭代模型 多次迭代完善 每次迭代是完整小周期，小步快跑 需求不确定、持续变化
原型模型 以原型导出需求 通过原型与用户交流、确认需求 需求难以表述、获取困难
螺旋模型 风险驱动的迭代 集成迭代与原型，强调风险分析 风险高、需求变化大的大型项目

选择原则：需求稳定选瀑布/增量；需求模糊易变选迭代/原型；风险高选螺旋。

题型：填空题考模型特点；简答题考「模型对比」「如何选择过程模型」。

SE4 软件需求基础

核心概念：

  • 需求工程：获取 → 分析 → 文档化 → 确认验证 → 管理的迭代过程。
  • 需求类型：
  • 功能需求：系统应提供的服务或功能。
  • 非功能需求：性能、可靠性、安全性、可用性等质量属性。
  • 领域需求：特定应用领域的业务规则。
  • 软件需求特征：正确性、一致性、完整性、可验证性、可追踪性、可修改性等。

题型：填空题考需求类型、需求工程活动；简答题考「需求工程过程」「良好需求应具备的特征」。

SE5 获取软件需求

核心活动：

  1. 明确问题及识别利益相关方（stakeholders）。
  2. 获取需求的方法：访谈、问卷、会议、现场观摩、业务资料分析、原型、头脑风暴等。
  3. 导出和构想软件需求。
  4. 描述初步的软件需求。
  5. 确认和验证初步软件需求。

题型：简答题考「需求获取方法有哪些」「需求获取步骤」。

SE6 分析软件需求

核心内容：

  • 从用例视角、行为视角、结构视角对需求建模。
  • 常用 UML 图：
  • 用例图：描述功能需求与参与者关系。
  • 顺序图 / 通信图：描述对象间交互的时间顺序 / 结构关系。
  • 类图：描述领域概念及关系。
  • 状态图：描述对象生命周期状态变化。
  • 用统一建模语言（UML）记录用户需求，用用例描述补充；交互过程可转化为顺序图；把需求汇总为类图。

题型：设计题重点，考用例图、顺序图、类图、状态图。

SE7 软件设计基础

核心内容：

  • 软件设计回答「如何做」，核心目标是满足需求并保证质量。
  • 设计内容：体系结构设计、接口设计、数据设计、过程/算法设计。
  • 设计原则：
  • 抽象、求精、模块化、信息隐藏、多视点、关注点分离。
  • 高内聚、低耦合。
  • 软件设计方法：
  • 结构化设计方法（面向数据流的设计方法学）。
  • 面向对象设计方法。
  • 结构化设计：把数据流图映射为软件结构图；常用 变换流 和 事务流。
  • 变换型：输入流 → 变换中心 → 输出流。
  • 事务型：事务中心 + 多条动作路径。

题型：简答题考「设计原则」「高内聚低耦合含义」；设计题考 DFD → 结构图。

SE10 软件详细设计

核心内容：

  • 详细设计在体系结构基础上精化模块内部：算法、数据结构、接口细节。
  • 常用工具：程序流程图、N-S 图、PAD 图、PDL、活动图。
  • 活动图要点：
  • 必须有 开始节点（start） 和 结束节点（end）。
  • 活动节点、判断/合并节点、分叉/汇合节点、泳道。

题型：设计题重点，考活动图绘制。

SE13 软件测试

核心内容：

  • 软件测试：通过运行测试用例发现缺陷，而不是证明无缺陷。
  • 测试过程：单元测试 → 集成测试 → 确认测试 → 系统测试 → 回归测试。
  • 测试方法：
  • 黑盒测试：依据外部功能，不需要了解代码；方法有等价类划分、边界值分析、决策表、因果图。
  • 白盒测试：依据内部结构，需要了解代码；方法有语句覆盖、分支覆盖、路径覆盖、基本路径测试。
  • 设计测试用例：
  • 等价类划分：有效等价类、无效等价类；针对有效等价类设计一个覆盖所有有效类的用例，无效等价类逐一不满足。
  • 边界值分析：取边界点及其左右邻近值，是最容易出错的用例。
  • 环形复杂度：$V(G) = E - N + 2 = P + 1$。

题型：简答题考「黑盒/白盒区别」「测试层次」；设计题考测试用例设计（等价类 + 边界值 + 路径覆盖）。

三、设计题 Step-by-Step 答题模板

设计题共 60 分，是考试核心。以下每类题型给出可复现的解题步骤。

3.1 用例图

步骤：

  1. 找 Actor：列出与系统交互的人、外部系统、硬件设备（如用户、管理员、银行系统）。
  2. 找 Use Case：把「可以做什么」转成动宾结构的功能（如查询订单、提交订单）。
  3. 画系统边界：用矩形框把所有用例框起来，框顶写系统名称。
  4. 建立关系：
    • 关联：Actor 与 Use Case 之间的实线。
    • <<include>>：某用例必然调用另一用例，箭头指向被包含用例。
    • <<extend>>：某用例在特定条件下扩展另一用例，箭头由扩展用例指向基用例。

检查清单：

  • [ ] Actor 放在系统边界外，Use Case 放在边界内。
  • [ ] 用例名称是动宾结构，不是名词。
  • [ ] include/extend 箭头方向正确。
  • [ ] 系统名称明确。

3.2 顺序图 / 通信图

顺序图步骤：

  1. 确定参与对象：从左到右排列，通常 Actor → 边界类 → 控制类 → 实体类 → 外部接口。
  2. 画生命线：每个对象向下画一条虚线。
  3. 按时间顺序画消息：
    • 同步消息：实心箭头。
    • 异步消息：开放箭头。
    • 返回消息：虚线箭头。
  4. 标激活条：对象执行活动期间画窄矩形。
  5. 处理分支：用 alt / opt / loop 组合片段表示选择、可选、循环。

通信图要点：强调对象间的结构关系，标注消息序号表示时间顺序。

3.3 类图

步骤：

  1. 找名词：从需求描述中列出所有候选类（人、物、事件、角色等）。
  2. 筛选有效类：去掉同义词、超出系统范围、属性误作类的名词。
  3. 确定属性与方法：属性是名词特征，方法是行为动词。
  4. 确定关系：
    • 关联：实线，可标多重性（1、*、1..*）。
    • 聚合：空心菱形，整体与部分，部分可独立存在。
    • 组合：实心菱形，整体与部分，部分随整体销毁。
    • 泛化（继承）：实线空心三角箭头，子类指向父类。
    • 依赖：虚线箭头。
  5. 画图并标注多重性。

检查清单：

  • [ ] 类名首字母大写，属性/方法可见性用 + - #。
  • [ ] 不要把属性当作独立类。
  • [ ] 继承箭头方向正确（子类 → 父类）。

3.4 状态图

步骤：

  1. 找对象生命周期：选择一个有明显状态变化的对象（如订单、电梯、线程）。
  2. 列状态：找出该对象可能出现的所有稳定状态。
  3. 找触发事件：找出导致状态转换的事件、动作、监护条件。
  4. 画图：
    • 初始状态：实心圆。
    • 终止状态：实心圆加空心圆。
    • 状态：圆角矩形。
    • 转换：带箭头的线，线上标注事件和 [条件]。

检查清单：

  • [ ] 状态是名词或形容词，不是动作。
  • [ ] 转换线上有触发事件。
  • [ ] 有初始状态和必要的终止状态。

3.5 活动图

步骤：

  1. 找活动：把流程中的动词短语列出来。
  2. 定顺序与分支：确定先后关系、判断条件、并行活动。
  3. 画泳道（可选）：按责任角色划分纵向区域。
  4. 画图：
    • 开始节点：实心圆。
    • 结束节点：同心圆。
    • 活动节点：圆角矩形。
    • 判断/合并节点：菱形。
    • 分叉/汇合节点：粗黑线。

检查清单：

  • [ ] 必须有 start 和 end。
  • [ ] 判断节点引出多条带条件的出边。
  • [ ] 并行活动用分叉/汇合表示。

3.6 DFD → 结构图

步骤 1：画数据流图

  1. 找外部实体（矩形）。
  2. 找加工（圆角矩形/圆圈），顶层 1 个总加工，下层按编号展开。
  3. 找数据存储（开口矩形/双横线）。
  4. 画数据流（带箭头的线，标注数据名称）。

步骤 2：判断数据流类型

  • 变换型：有明显的输入流 → 变换中心 → 输出流。
  • 事务型：有一个事务中心，根据事务类型分发到多条动作路径。

步骤 3：映射为结构图

  • 变换型：传入路径模块 → 变换控制模块 → 传出路径模块。
  • 事务型：接收路径控制模块 → 事务分析/散转模块 → 各动作路径模块。

3.7 程序流程图 + 环形复杂度 + 路径覆盖

步骤 1：画程序流程图

  • 用圆角矩形/矩形表示处理，菱形表示判断，箭头表示控制流向。

步骤 2：转换为控制流图

  • 每个顺序执行的基本块作为一个节点。
  • 判断节点引出多条有向边。

步骤 3：计算环形复杂度

$$ V(G) = E - N + 2 $$

或 $V(G) = P + 1$，其中 $P$ 为判定节点数。

步骤 4：确定基本路径集

  • 根据复杂度确定独立路径数量。
  • 每条路径至少包含一条其他路径未走过的边。

步骤 5：设计测试用例

  • 每条路径设计一个测试用例，给出输入、执行路径、预期输出。

3.8 黑盒测试用例设计

步骤 1：等价类划分

  • 有效等价类：符合规格说明的合理输入集合。
  • 无效等价类：不符合规格说明的不合理输入集合。
  • 常见类型：范围、特定值、集合、布尔量。

步骤 2：边界值分析

  • 范围 $(a,b)$：取 $a$、$b$ 及紧邻左右的值。
  • 一组数：取最大、最小、次大、次小。
  • 数据结构有界时检查其边界。

步骤 3：写测试用例

用例编号 输入数据 所属等价类 预期输出
TC01 ... 有效类 ...
TC02 ... 无效类 ...

要点：

  • 一个有效用例尽量覆盖多个有效等价类。
  • 每个无效等价类单独设计一个用例。
  • 边界值是最容易出错的用例，必须包含。

3.9 白盒测试用例设计

常见覆盖标准：

覆盖标准 要求 用例设计要点
语句覆盖 每条语句至少执行一次 最简单，但发现错误能力弱
分支覆盖 每个判断的真/假分支都执行 比语句覆盖强
路径覆盖 每条可能路径都执行 最彻底，但路径数可能爆炸
基本路径覆盖 覆盖所有独立路径 用环形复杂度确定路径数，实用且足够

步骤：

  1. 画控制流图。
  2. 计算 $V(G)$ 确定独立路径数。
  3. 列出基本路径。
  4. 为每条路径设计输入和预期输出。

四、简答题万能模板

4.1 概念类题目

模板：

定义：XXX 是指在 ... 过程中 ... 的一种 ...。 核心要点：① ...；② ...；③ ...。 举例 / 作用：例如 ...，其目的是 ...。

示例：什么是软件危机？

软件危机是指在计算机软件开发和维护过程中遇到的一系列严重问题。主要表现为开发成本高、进度难以控制、质量难以保证、维护困难、失败风险大。原因是早期对软件复杂性和特殊性认识不清、缺乏有效开发方法和成功经验。

4.2 对比类题目

模板：

从定义、关注点、依据、方法、阶段、优缺点等维度列表对比。

示例：黑盒测试与白盒测试的区别。

维度 黑盒测试 白盒测试
依据 外部功能与性能描述 程序内部结构
是否需了解代码 不需要 需要
测试对象 功能、接口、输入输出 语句、分支、路径
典型方法 等价类划分、边界值分析 语句覆盖、分支覆盖、路径覆盖
典型阶段 集成测试、确认测试、系统测试 单元测试

4.3 过程类题目

模板：

某过程包括以下 $n$ 个阶段： 1. 阶段一名称：核心任务 ...，主要产出 ...。 2. 阶段二名称：核心任务 ...，主要产出 ...。 ... 各阶段之间存在迭代/回溯关系。

示例：简述软件生命周期各阶段任务。

软件生命周期包括需求分析、概要设计、详细设计、编码实现、集成测试、确认测试等阶段。需求分析定义功能与非功能需求；概要设计建立总体架构；详细设计设计模块内部细节；编码实现编写代码并单元测试；集成测试组装模块；确认测试从用户角度验证需求。

五、填空题高频速记

5.1 定义与组成

  • 软件 = 程序 + 数据 + 文档
  • 软件开发 = 软件创作 + 软件生产
  • 软件工程三要素 = 过程 + 方法 + 工具
  • 软件危机主要表现为：开发成本高、进度难以控制、质量难以保证、维护困难、失败风险大
  • 软件工程诞生于 1968 年 NATO 会议

5.2 需求

  • 需求类型：功能需求、非功能需求、领域需求
  • 需求工程过程：获取 → 分析 → 文档化 → 确认验证 → 管理
  • 需求获取方法：访谈、问卷、会议、现场观摩、业务资料分析、原型、头脑风暴
  • 需求分析三视角：用例视角、行为视角、结构视角

5.3 设计原则

  • 软件设计核心目标：满足需求并保证质量
  • 模块设计原则：高内聚、低耦合
  • 常用设计原则：抽象、求精、模块化、信息隐藏、多视点、关注点分离
  • 结构化设计方法学是面向数据流的设计方法学
  • 常用数据流类型：变换流、事务流

5.4 测试

  • 测试目的：发现缺陷，不是证明无缺陷
  • 测试层次：单元测试 → 集成测试 → 确认测试 → 系统测试 → 回归测试
  • 黑盒测试方法：等价类划分、边界值分析、决策表、因果图
  • 白盒测试方法：语句覆盖、分支覆盖、路径覆盖、基本路径测试
  • 确认测试包括 α 测试（开发方内部模拟用户）和 β 测试（典型用户实际环境使用）
  • 集成策略：自顶向下、自底向上
  • 单元测试需要 驱动程序 和 桩模块

5.5 UML 图

  • 用例图描述 功能需求与参与者关系
  • 顺序图强调消息传递的 时间顺序
  • 通信图强调对象间的 合作关系与结构
  • 类图描述 领域概念及相互关系
  • 状态图描述 对象生命周期状态变化
  • 活动图描述 业务流程或操作控制流

六、必背公式与定义

6.1 软件 = 程序 + 数据 + 文档

$$ \text{软件} = \text{程序} + \text{数据} + \text{文档} $$

6.2 测试用例四元组

$$ (\text{输入数据},\ \text{前置条件},\ \text{测试步骤},\ \text{预期输出}) $$

6.3 环形复杂度

$$ V(G) = E - N + 2 $$

其中 $E$ 为流图边数，$N$ 为流图节点数；或简化为 $V(G) = P + 1$，$P$ 为判定节点数。

6.4 维护工作量模型

$$ M = P + K \cdot e^{(c-d)} $$

其中 $P$ 为生产性工作量，$K$ 为经验常数，$c$ 为复杂度，$d$ 为熟悉程度。

6.5 缺陷链

缺陷 → 错误 → 失效

6.6 面向对象设计原则

SOLID + 迪米特法则：

  • S：单一职责原则（Single Responsibility Principle）
  • O：开闭原则（Open/Closed Principle）
  • L：里氏替换原则（Liskov Substitution Principle）
  • I：接口隔离原则（Interface Segregation Principle）
  • D：依赖倒置原则（Dependency Inversion Principle）
  • LoD：迪米特法则 / 最少知识原则（Law of Demeter）

七、高频考点与应试提示

7.1 高频简答题

  1. 软件危机：在计算机软件开发和维护过程中遇到的一系列严重问题；表现为开发成本高、进度难以控制、质量难以保证、维护困难、失败风险大；原因是对软件复杂性和特殊性认识不清、缺乏有效开发方法、缺乏成功经验。
  2. 软件生命周期各阶段任务：需求分析、概要设计、详细设计、编码实现、集成测试、确认测试。
  3. 软件工程三要素：过程、方法、工具；过程定义活动顺序，方法提供技术手段，工具提高效率。
  4. 顺序图与通信图区别：二者同属 UML 交互图、语义等价、可相互转换；顺序图强调消息传递的时间顺序，通信图强调对象间的合作关系与结构。
  5. 模块独立性度量：内聚度（模块内部结合紧密程度，越高越好）和耦合度（模块之间相关程度，越低越好）；设计原则为高内聚、低耦合。
  6. 测试步骤及对象：单元测试（模块内部）、集成测试（模块接口）、确认测试（功能与性能）、系统测试（整体系统）、回归测试（修改后的版本）。
  7. 黑盒测试与白盒测试区别：见 4.2 对比模板。

7.2 高频设计题

  1. 画用例图：明确 Actor、Use Case、系统边界、include/extend/泛化关系。
  2. 画顺序图：对象从左到右依次为 Actor → 边界类 → 控制类 → 实体类 → 外部接口；消息按时间自上而下传递，注意生命线、激活期、同步/异步/返回消息图符。
  3. 画类图：找类 → 定属性方法 → 定关系 → 标多重性；注意不要把属性当类，继承箭头方向正确。
  4. 画 DFD 并转化为结构图：先画 0 层和 1 层 DFD；区分变换型与事务型数据流图；映射为层次结构图。
  5. 程序流程图 + 环形复杂度 + 路径覆盖：正确绘制程序流程图，转换为控制流图，计算 $V(G)$，根据基本路径设计测试用例。
  6. 等价类划分 + 边界值分析：按范围、特定值、集合、布尔量划分有效/无效等价类；边界值重点选取边界点及其左右邻近值；给出输入、预期输出。

7.3 应试提示

  • 先审题再画图：看清题目要求结构化方法还是面向对象方法，画几层图，是否要求转化为结构图。
  • 图符规范：UML 图符（Actor 火柴人、用例椭圆、类三层矩形、菱形聚合/组合、空心三角继承等）要画对。
  • 设计题步骤分：即使图画不完全，也要把分析步骤写清楚，按步骤给分。
  • 路径覆盖用例：每个用例需给出输入、执行路径、预期输出，避免只写输入。
  • 对比题：从「定义、关注点、依据、方法、阶段、优缺点」等维度分条作答。
  • 不要混淆概念：
  • 增量模型是「一块一块地做」，迭代模型是「一遍一遍地改」。
  • 测试的目的是发现缺陷，不是证明无缺陷。
  • 软件不会物理磨损，但会逻辑老化。

提示：本文档为考前速查，建议对照教材课件中的各类示意图进行手绘练习，确保能在考场上独立完成数据流图、结构图、用例图、顺序图、类图、状态图、活动图和程序流程图的绘制。
